title: Training the vocabulary
description: The word-count plus incremental pair-index algorithm, its complexity, and its determinism
---

Training a BPE tokenizer means discovering, in order, the most useful byte
merges from a corpus. The textbook algorithm is simple to state and painfully
slow to run at scale. `tinyllm`'s trainer (`BPETokenizer.train`) keeps the exact
same output but makes it tractable — 32,511 merges over roughly a gigabyte of
text, in pure Python — with two ideas: **count unique words once**, and
**maintain the pair counts incrementally** instead of rescanning.

## The naive algorithm and why it hurts

Classic BPE, done literally:

1. Encode the whole corpus as a flat sequence of bytes.
2. Count every adjacent pair.
3. Merge the most frequent pair everywhere it occurs.
4. Go to step 2 and repeat until you have enough merges.

The problem is step 2. Recounting all adjacent pairs across the entire corpus,
once per merge, is `O(n_merges × corpus_length)`. With ~32k merges and a
billion-token corpus that is tens of trillions of operations — a non-starter in
Python.

`tinyllm` attacks both factors: it shrinks `corpus_length` down to the set of
*distinct words*, and it removes the "once per merge" rescan entirely.

## First win: count unique words, not the corpus

Because merges are confined to a single pre-token, two occurrences of the same
word are indistinguishable to BPE. So the corpus is collapsed to a
**word-to-frequency** table before any merging happens. Natural-language word
frequencies are Zipfian: a corpus of a billion tokens has only on the order of
millions of *distinct* pre-tokens, and the bulk of the text is a small number of
very common words. Deduplicating turns "process the corpus" into "process each
unique word once, weighted by how often it appears."

```python
word_counts: Counter[bytes] = Counter()
if num_proc > 1:
    with Pool(num_proc) as pool:
        for c in pool.imap_unordered(_count_words, texts, chunksize=64):
            word_counts.update(c)
else:
    for t in texts:
        word_counts.update(_count_words(t))
```

`_count_words` pre-tokenizes one document with the split regex and returns a
`Counter` of the UTF-8 bytes of each chunk. Counting is embarrassingly parallel
across documents, so it optionally fans out over a process `Pool` and merges the
partial counters. Every subsequent operation carries a frequency weight `c`, so
merging a pair inside a word that appears a million times counts as a million
occurrences without ever touching a million copies.

### Pruning with `min_word_freq`

Web text has a long tail of junk pre-tokens: typos, mojibake, one-off Unicode
soup. Before training, words below a frequency floor can be dropped:

```python
if min_word_freq > 1:
    word_counts = Counter(
        {w: c for w, c in word_counts.items() if c >= min_word_freq}
    )
```

The default is `1` (keep everything). Raising it discards rare pre-tokens, which
both speeds up training (fewer distinct words to maintain) and steers the merge
budget toward patterns that actually recur, rather than memorizing noise.

The canonical 32k tokenizer does raise it: `scripts/train_tokenizer.py` declares
`--min-word-freq` with a default of `2`, so any pre-token seen exactly once in
the ~250 MB training sample is dropped before the first merge.

## Second win: the incremental pair-count index

After deduplication, each word is a mutable list of ids and carries its count.
Two structures are built once and then maintained in place:

```python
pair_counts: Counter[tuple[int, int]] = Counter()
pair_to_words: dict[tuple[int, int], set[int]] = {}
for wi, w in enumerate(words):
    c = counts[wi]
    for pair in zip(w, w[1:]):
        pair_counts[pair] += c
        pair_to_words.setdefault(pair, set()).add(wi)
```

- **`pair_counts`** maps each adjacent id pair to its total frequency-weighted
  count across all words. This is the table we pick the winner from.
- **`pair_to_words`** is an inverted index: for each pair, the set of word
  indices that currently contain it. This is the key to never rescanning — when
  a pair wins, we only need to visit the handful of words that actually contain
  it.

Each merge step then does local surgery:

```python
for step in range(n_merges):
    if not pair_counts:
        break
    # highest count wins; ties broken by smallest pair ids => deterministic
    best = max(pair_counts.items(), key=lambda kv: (kv[1], (-kv[0][0], -kv[0][1])))[0]
    new_id = 256 + step
    merges.append(best)
    for wi in list(pair_to_words.get(best, ())):
        w, c = words[wi], counts[wi]
        for pair in zip(w, w[1:]):
            pair_counts[pair] -= c
            if pair_counts[pair] <= 0:
                del pair_counts[pair]
            s = pair_to_words.get(pair)
            if s is not None:
                s.discard(wi)
        merged: list[int] = []
        i = 0
        while i < len(w):
            if i < len(w) - 1 and w[i] == best[0] and w[i + 1] == best[1]:
                merged.append(new_id)
                i += 2
            else:
                merged.append(w[i])
                i += 1
        words[wi] = merged
        for pair in zip(merged, merged[1:]):
            pair_counts[pair] += c
            pair_to_words.setdefault(pair, set()).add(wi)
```

The step reads top to bottom as: pick the winning pair; for every word that
contains it, first **subtract** that word's old pairs out of the global counts
and drop it from their inverted-index sets, then **rewrite** the word by
replacing every occurrence of the winning pair with the new id, then **add** the
word's new pairs back into the counts and the index. The global `pair_counts`
and `pair_to_words` end the step exactly as if we had recounted from scratch —
but we only ever touched words that contained the merged pair.

## Complexity

Let `W` be the number of distinct words and `P` the number of live distinct
pairs at a given moment.

- **Finding the winner** is `max(...)` over `pair_counts`, i.e. `O(P)` per step.
  This linear scan is the dominant per-step cost, and it is over the compact set
  of distinct pairs, not the corpus.
- **Applying the merge** costs work proportional only to the total length of the
  words that contained the winning pair — decrement, rewrite, re-increment. In
  aggregate over the whole run, each pair-occurrence is created and destroyed a
  bounded number of times, so applied-merge work is roughly linear in the total
  number of pair-occurrences ever present, independent of `n_merges`.

The naive `O(n_merges × corpus_length)` becomes, in practice, dominated by
`O(n_merges × P)` for winner selection over a deduplicated, incrementally
maintained state. That is the difference between "runs overnight in Python" and
"never finishes."

## Determinism: how ties are broken

Reproducibility matters — the same corpus must always yield the same tokenizer,
regardless of hash-set iteration order or how many processes counted the words.
That is guaranteed entirely by the sort key when choosing the winner:

```python
best = max(pair_counts.items(), key=lambda kv: (kv[1], (-kv[0][0], -kv[0][1])))[0]
```

`max` compares tuples `(count, (-a, -b))` for a candidate pair `(a, b)`:

- **Primary key: `count`.** The most frequent pair wins, as expected.
- **Tie-break: `(-a, -b)`.** Among pairs tied on count, `max` prefers the
  largest `(-a, -b)`, which means the *smallest* `a`, and then the smallest `b`.
  So ties resolve to the pair with the smallest first id, then the smallest
  second id.

Because both keys are total orders over the pair, there is never ambiguity: the
winner is a pure function of the current counts, never of dictionary or set
ordering. `pair_to_words` sets are only ever iterated to *apply* a chosen
merge, never to *choose* one, so their unordered nature cannot leak into the
result. Two runs on identical input produce byte-identical merge lists.

## Encoding replays the merges by rank

Training produces an ordered merge list; encoding new text replays it. The
merge order is stored as a **rank** table (`merge_ranks`: pair to its index),
and each pre-token is encoded greedily by always applying the lowest-rank
applicable merge first:

```python
def _encode_word_uncached(self, word: bytes) -> tuple[int, ...]:
    ids = list(word)
    while len(ids) >= 2:
        pairs = set(zip(ids, ids[1:]))
        best = min(pairs, key=lambda p: self.merge_ranks.get(p, float("inf")))
        if best not in self.merge_ranks:
            break
        new_id = 256 + self.merge_ranks[best]
        out: list[int] = []
        i = 0
        while i < len(ids):
            if i < len(ids) - 1 and ids[i] == best[0] and ids[i + 1] == best[1]:
                out.append(new_id)
                i += 2
            else:
                out.append(ids[i])
                i += 1
        ids = out
    return tuple(ids)
```

Any pair not in the rank table gets rank infinity and is never chosen, so
encoding stops when no learned merge applies — leaving whatever bytes remain as
individual tokens. Per-word results are memoized with an `lru_cache` of 65,536
entries (`1 << 16`), so the Zipfian flood of common words is encoded once and
served from cache thereafter.

`encode` then just glues the per-word results together in pre-token order:

```python
def encode(self, text: str) -> list[int]:
    """Encode text to token ids. Special tokens are never produced."""
    ids: list[int] = []
    for m in _PATTERN.finditer(text):
        ids.extend(self._encode_word(m.group().encode("utf-8")))
    return ids
```
