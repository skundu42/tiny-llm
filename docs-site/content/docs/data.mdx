Once text is tokenized it becomes a river of integers, and the training loop
needs to drink from that river efficiently, reproducibly, and without loading it
all into memory. `tinylm`'s data path (`tinylm/data.py` plus
`scripts/prepare_data.py`) is three ideas: a compact on-disk **shard format**, a
**memory-mapped random-crop sampler**, and a **stateless RNG** that turns
"resume training exactly where it left off" from a hard problem into a non-issue.

## Why FineWeb-Edu

The pretraining corpus is `HuggingFaceFW/fineweb-edu` (the `sample-10BT`
subset). FineWeb-Edu is a quality-filtered educational slice of FineWeb, itself
a cleaned and deduplicated distillation of Common Crawl. The selling point is
signal density: an educational-quality classifier keeps the explanatory,
factual, well-formed pages and drops the SEO spam, boilerplate, and link farms
that make up much of the raw web.

That trade matters more the smaller the model. At a ~10-billion-token budget you
cannot out-scale bad data: the model sees each concept only a handful of times,
so every token should carry as much clean signal as possible. Quality beats
quantity at this scale, the subset is openly available, and it streams, so
`prepare_data.py` can consume it without ever downloading the whole thing:

```python
ds = load_dataset(args.dataset, name=args.dataset_config or None,
                  split=args.split, streaming=True)
docs = (ex[args.text_key] for ex in ds)
```

The dataset coordinates are CLI flags with FineWeb-Edu defaults
(`--dataset HuggingFaceFW/fineweb-edu --dataset-config sample-10BT`), so any
HuggingFace text corpus can stand in; see
[Train on a custom dataset](/docs/runbooks/custom-dataset).

## Shard format: uint16 bins plus an index

Tokenized documents are written as fixed-size binary shards by `ShardWriter`.
Each shard is a flat array of token ids in **`uint16`**:

```python
self.buffer = np.empty(shard_tokens, dtype=np.uint16)
```

`uint16` is the whole game for on-disk size. The vocabulary is 32,768 tokens
(see [Tokenizer](/docs/tokenizer)), which fits in 16 bits with room to spare, so
each token costs exactly **two bytes**, half of what a 32-bit format would use.
A 10-billion-token corpus is therefore ~20 GB on disk instead of ~40 GB, and the
shards are raw contiguous arrays with no per-token overhead. The writer buffers
up to `shard_tokens` ids, then flushes a numbered `.bin` file straight to disk:

```python
def _flush(self) -> None:
    if self.fill == 0:
        return
    name = f"{self.split}_{len(self.entries):05d}.bin"
    self.buffer[: self.fill].tofile(os.path.join(self.out_dir, name))
    self.entries.append({"file": name, "tokens": int(self.fill)})
    self.fill = 0
```

Alongside the shards, a single **`index.json`** records the layout: the dtype
and, per split, the ordered list of shard files with their token counts:

```json
{
  "dtype": "uint16",
  "splits": {
    "train": [{"file": "train_00000.bin", "tokens": 100000000}, ...],
    "val":   [{"file": "val_00000.bin", "tokens": 10000000}]
  }
}
```

Two details are worth calling out. First, the index is written **atomically**
(to a `.tmp` file, then `os.replace`), so a crash mid-write can never leave a
half-written index that would corrupt every future read:

```python
tmp = path + ".tmp"
with open(tmp, "w") as f:
    json.dump(index, f, indent=1)
os.replace(tmp, path)
```

Second, `ShardWriter` refuses to re-open a split that already exists in the
index, so you cannot accidentally append a second `train` on top of an old one;
you get a `ValueError` telling you to use a fresh directory.

### Documents and the end-of-text separator

`ShardWriter` itself is format-agnostic: it appends whatever ids it is handed and
slices them into fixed shards, spanning shard boundaries freely. The document
structure is imposed by the caller. `prepare_data.py` encodes each document and
appends the `<|endoftext|>` id before writing:

```python
for ids in encs:
    ids.append(eot)
    w = val_w if val_w.total_written < args.val_tokens else train_w
    w.write(ids)
```

So the token stream is `doc | eot | doc | eot | ...`. The very first
`val_tokens` tokens are routed to the `val` split and the remainder to `train`,
giving a clean held-out set carved from the same distribution. Bulk encoding uses
the verified fast tokenizer (`fast.encode_batch`); the pure-Python encoder is
the fallback (`--slow`). See
[the tokenizer fast path](/docs/tokenizer/format-and-fast-path) for the parity
check that gates it.

## Sampling: memmapped random crops

Training does not read shards start to finish. It draws random fixed-length
windows ("crops") from anywhere in the data. `TokenShards` opens every shard of a
split as a **memory map**:

```python
self.shards = [
    np.memmap(os.path.join(data_dir, e["file"]), dtype=np.uint16, mode="r")
    for e in entries
]
```

`np.memmap` is what lets a 20 GB dataset train inside a few GB of RAM: nothing is
loaded up front. The OS pages shard bytes in on demand as crops touch them and
evicts cold pages under memory pressure, so the working set is roughly "the crops
in flight," not "the whole corpus."

`sample_batch` draws a batch of crops:

```python
def sample_batch(self, batch_size: int, seq_len: int, rng: np.random.Generator,
                 device: str | torch.device = "cpu"):
    usable = self.sizes - (seq_len + 1)
    positions = np.clip(usable + 1, 0, None)  # count of valid crop offsets per shard
    assert positions.sum() > 0, "no shard is long enough for this seq_len"
    p = positions.astype(np.float64) / positions.sum()
    shard_ids = rng.choice(len(self.shards), size=batch_size, p=p)
    xs = np.empty((batch_size, seq_len + 1), dtype=np.int64)
    for i, si in enumerate(shard_ids):
        off = int(rng.integers(0, positions[si]))
        xs[i] = self.shards[si][off : off + seq_len + 1].astype(np.int64)
    t = torch.from_numpy(xs)
    x, y = t[:, :-1].contiguous(), t[:, 1:].contiguous()
    if str(device) != "cpu":
        x = x.to(device, non_blocking=True)
        y = y.to(device, non_blocking=True)
    return x, y
```

Read it in three moves:

1. **Count valid crop positions per shard.** A crop needs `seq_len + 1` tokens
   (one extra for the shifted target). A shard of length `L` admits start offsets
   `0 .. L - (seq_len + 1)`, i.e. `L - seq_len` distinct positions, which is
   exactly `positions`. Shards too short to hold a single crop clip to `0` and
   are simply never selected.

2. **Sample weighted by valid positions, not uniformly over shards.** The shard
   is chosen with probability `positions / positions.sum()`. This is the subtle
   correctness point: weighting by the number of valid offsets makes **every
   crop position across the entire dataset equally likely**. Choosing a shard
   uniformly instead would over-sample tokens that live in small shards: a
   100M-token shard and a 10M-token shard would each get picked half the time,
   badly skewing the mix. Position-weighting makes sampling uniform over the
   token stream, which is what you actually want.

3. **Cut, shift, and split.** A uniform offset within the chosen shard reads
   `seq_len + 1` contiguous ids (cast to `int64` for the embedding lookup).
   Slicing `[:, :-1]` and `[:, 1:]` yields the input `x` and the next-token
   target `y`, offset by one: the standard causal-LM setup. On GPU the transfer
   is `non_blocking` to overlap with compute.

Sampling is with replacement and independent per draw. There is no epoch
boundary and no shuffle buffer: crops may overlap or repeat, which over billions
of tokens is entirely fine and is standard for LLM pretraining. Crops can also
straddle an `<|endoftext|>`, so the model learns document boundaries as ordinary
in-context events.

## The stateless RNG, and why resume is trivial

Here is the design decision that pays off everywhere else. The sampler holds no
random state of its own. The training loop constructs a **fresh** generator for
every micro-batch, seeded by exactly which micro-batch it is:

```python
rng = np.random.default_rng([tc.seed, rank, step, micro])
x, y = train_data.sample_batch(tc.micro_batch_size, mc.seq_len, rng, device)
```

The batch drawn at `(seed, rank, step, micro)` is a **pure function** of that
tuple. NumPy's `default_rng` derives a well-distributed, independent PCG64 stream
from the seed sequence, so each `(step, micro)` gets its own uncorrelated draw,
and nothing carries over from one step to the next. Four consequences fall out:

- **Resume needs to save only a step number.** There is no dataloader iterator,
  no shuffle buffer, no epoch pointer, no sampler position to serialize. To
  resume, restore the weights and optimizer state, set `start_step`, and the loop
  regenerates the *identical* token stream from that step forward, as if it had
  never stopped. Contrast this with a stateful shuffling `DataLoader`, where
  resuming mid-epoch means faithfully restoring the sampler's internal state,
  something frameworks often get subtly wrong or skip entirely.

- **Each rank sees different data, deterministically.** Because `rank` is in the
  seed, the eight replicas of a data-parallel run make independent deterministic
  draws. Sampling is with replacement, so rare cross-rank overlaps are possible;
  the global batch still has `world × micro_batch_size × grad_accum` crops and is
  perfectly reproducible for a fixed world size.

- **Validation lives in its own namespace.** Single-process evaluation seeds as
  `np.random.default_rng([tc.seed, 999, k])`; DDP additionally includes `rank`.
  The validation sample stays constant across evaluations, making val-loss
  comparisons apples to apples. Seed separation makes it independent of the
  training draws, though random crops can still coincide by chance.

- **One caveat: the world size is baked in.** Because `rank` and `grad_accum`
  shape which crops are drawn, resuming on a *different* number of GPUs changes
  the data stream. Bit-exact resume assumes the same world size; see
  [Checkpointing](/docs/training/checkpointing) for the full reproducibility
  guarantee and the test that proves it.

This is the quiet foundation under the whole training story: because the data at
any point is fully determined by `(seed, rank, step, micro)`, a checkpoint only
has to capture weights, optimizer state, and a step counter to reproduce the
future bit for bit.
