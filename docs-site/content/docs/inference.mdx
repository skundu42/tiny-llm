Generation is where the model stops being a function of a whole sequence and
becomes a loop. `TinyLM.generate` in `tinylm/model.py` is that loop: allocate a
cache, run the prompt through once, then emit one token at a time until the
budget or an end-of-text token stops it. This page covers the contract that loop
obeys, what the cache costs in bytes, and how the three sampling knobs compose.

## The whole loop

```python
@torch.no_grad()
def generate(self, idx, max_new_tokens, temperature=1.0, top_k=None, top_p=None, eos_id=None):
    # Validate shape, length, temperature/top-k/top-p, and eos_id up front.
    ...
    B, T = idx.shape
    total = T + max_new_tokens
    if total > self.cfg.seq_len:
        raise ValueError("generation would exceed context length")
    if max_new_tokens == 0:
        return idx

    was_training = self.training
    self.eval()
    try:
        cache = KVCache(...)
        logits, _ = self(idx, cache=cache)
        finished = torch.zeros(B, dtype=torch.bool, device=idx.device)
        for token_idx in range(max_new_tokens):
            # temperature -> top-k -> top-p -> sample
            ...
            if eos_id is not None:
                next_tok = torch.where(
                    finished[:, None], torch.full_like(next_tok, eos_id), next_tok
                )
                finished |= next_tok.squeeze(1).eq(eos_id)
            idx = torch.cat([idx, next_tok], dim=1)
            if eos_id is not None and finished.all():
                break
            if token_idx + 1 < max_new_tokens:
                logits, _ = self(next_tok, cache=cache)
        return idx
    finally:
        self.train(was_training)
```

Three structural decisions are visible before any sampling happens:

- **Inputs and length are checked up front.** Invalid sampling controls raise a
  descriptive `ValueError`, and `total <= self.cfg.seq_len` is enforced
  *before* a single token is generated. Prompt plus new tokens must fit the
  context: there is no sliding window, no rolling eviction, no truncation. At
  `d26`'s `seq_len = 2048`, a 1,900-token prompt leaves you 148 tokens. The assert
  is the honest version of a limit that exists either way: the
  [RoPE tables](/docs/architecture/rope) are precomputed to exactly `seq_len`
  rows, so position 2048 has no rotation to apply.
- **The cache is allocated once, for exactly `total`.** Not `seq_len`: `T +
  max_new_tokens`. Generating 50 tokens from a 10-token prompt allocates a
  60-position cache, not a 2048-position one. Preallocating the true bound means
  no reallocation and no `torch.cat` in the hot loop.
- **The cache dtype is `self.embed.weight.dtype`**: it follows the model's own
  precision rather than being hardcoded. This has a real consequence covered
  [below](#a-note-on-dtype).

## The prefill/decode contract

The loop runs the model in exactly two modes, and the split is not an
optimization detail; it is a correctness contract the code asserts.

**Prefill.** `logits, _ = self(idx, cache=cache)`: one forward over the entire
prompt, from an empty cache (`pos == 0`). All `T` prompt positions are computed
in parallel, their keys and values written to the cache, causally masked.

**Decode.** `logits, _ = self(next_tok, cache=cache)`: one forward per token,
`T == 1`, appending to a non-empty cache.

`Attention.forward` permits those two cases and nothing else:

```python
if cache is not None:
    assert T == 1 or cache.pos == 0, (
        "chunked prefill unsupported: q_len > 1 requires an empty cache"
    )
    k, v = cache.update(self.layer_idx, k, v)
```

### Why chunked prefill would leak the future

This is worth spelling out, because the assert looks like a performance guard and
is actually a correctness one. Recall how the mask is chosen:

```python
y = F.scaled_dot_product_attention(q, k, v, is_causal=q.size(2) == k.size(2))
```

`is_causal` is **derived**, not passed. Trace the two legal cases:

| Case | `q_len` | `k_len` | `is_causal` | Correct? |
|---|---|---|---|---|
| Prefill from empty | `T` | `T` | `True` | Yes: triangular mask applied |
| Decode one token | `1` | `pos + 1` | `False` | Yes: the one query is newest; all cached keys are its past |
| **Chunked prefill** | `5` | `105` | **`False`** | **No: silent corruption** |

In that third row (feeding 5 new tokens against 100 cached ones), `q_len !=
k_len`, so `is_causal` is `False` and **no mask is applied at all**. The 5 queries
would attend to each other bidirectionally: chunk token 1 would see chunk tokens
2-5, which are its future. Nothing would crash. The logits would just be wrong,
computed from information the model cannot have at inference time.

Supporting it properly needs a block-diagonal mask (causal within the chunk,
fully visible over the cached prefix), which SDPA will not derive for you.
Rather than build that, the code refuses the case loudly and documents the
refusal in `KVCache`'s own docstring:

```python
class KVCache:
    """Preallocated per-layer K/V cache for autoregressive decoding. Usage
    contract: one multi-token prefill from an empty cache (pos == 0), then
    exactly one token per forward call. A multi-token forward with pos > 0
    would attend without any causal mask and leak future tokens within the
    chunk; Attention asserts against it.
    """
```

**This is a real limitation, and it has real costs.** You cannot prefill a long
prompt in memory-bounded chunks, and you cannot extend a cached conversation with
a multi-token user turn in one pass; you must decode it token by token, at
1/`n` the throughput. Both are things a production server needs. The trade made
here is explicit: the two supported patterns are *provably* correct and cover
everything `generate` does, and a wrong mask is a bug that produces
plausible-looking output instead of an error. An assert that fires is strictly
better than a model that quietly attends to the future.

### The cache itself

```python
def update(self, layer: int, k: torch.Tensor, v: torch.Tensor):
    t = k.size(2)
    self.k[layer, :, :, self.pos : self.pos + t] = k
    self.v[layer, :, :, self.pos : self.pos + t] = v
    return self.k[layer, :, :, : self.pos + t], self.v[layer, :, :, : self.pos + t]

def advance(self, t: int) -> None:
    self.pos += t
```

`update` writes into the preallocated slice at `pos` and returns a **view** of
everything up to `pos + t`: no copy, no concatenation, no growth. Note the keys
are cached **after** RoPE and QK-norm, in their final rotated form; a key's
rotation depends only on its own absolute position, so it is valid forever.

`advance` is called once per forward by `TinyLM.forward`, after every block has
written, not once per layer, since all 26 layers share one position counter:

```python
def forward(self, idx, targets=None, cache: KVCache | None = None):
    B, T = idx.shape
    pos0 = cache.pos if cache is not None else 0
    cos = self.rope_cos[pos0 : pos0 + T]
    sin = self.rope_sin[pos0 : pos0 + T]
    x = self.embed(idx)
    for block in self.blocks:
        x = block(x, cos, sin, cache)
    if cache is not None:
        cache.advance(T)
```

`pos0` is where RoPE picks up: the cache's position *is* the absolute position of
the incoming token, so the decode step at `pos == 41` slices `rope_cos[41:42]`.
This is the coupling that makes the cache correct, and the reason `advance` must
come after the blocks, never before.

### The prefill logits shortcut

One easily-missed optimization at the end of `forward`:

```python
if cache is not None:
    return self.lm_head(x[:, -1:, :]), None
return self.lm_head(x), None
```

When a cache is in play, **only the last position's logits are computed**.
`lm_head` is the single widest matmul in the model (`d_model × vocab_size =
1280 × 32,768 = 41.9M` parameters), and during prefill you need exactly one row
of its output: the distribution for the token that follows the prompt. Projecting
all `T` prompt positions to vocab would cost `T×` as much for output that is
thrown away. On a 1,000-token prompt this alone is the difference between one
41.9M-parameter matmul and a thousand of them.

(The `targets is not None` branch above it returns full logits: training needs a
prediction at every position.)

## KV-cache memory

The cache is one preallocated tensor pair per layer, so its size is exact
arithmetic rather than an estimate:

```
bytes = 2 (K and V) × n_layer × batch × n_kv_head × seq_len × head_dim × bytes_per_element
```

For `d26` (`n_layer = 26`, `n_kv_head = 4`, `head_dim = 64`) at full context in
bf16, batch 1:

```
2 × 26 × 1 × 4 × 2048 × 64 × 2 = 54,525,952 bytes = 52 MiB
```

The decisive term is `n_kv_head`. [GQA](/docs/architecture/attention#gqa-why-4-kv-heads)
uses 4 KV heads against 20 query heads; without it, `n_kv_head` would be 20 and
the same formula gives:

```
2 × 26 × 1 × 20 × 2048 × 64 × 2 = 272,629,760 bytes = 260 MiB
```

**Exactly 5× larger**: the ratio `20/4`, since every other term is unchanged.

| Configuration | Batch 1 | Batch 32 |
|---|---|---|
| `d26` (4 KV heads, bf16) | **52 MiB** | **1.63 GiB** |
| Without GQA (20 KV heads, bf16) | 260 MiB | 8.13 GiB |
| `smoke` (6 layers, 2 KV heads, ctx 512) | 1.5 MiB | 48 MiB |

Every term is linear, which is what makes the cache the binding constraint on a
serving box. The **weights are a fixed cost**: 489,297,408 params × 2 bytes =
933 MiB in bf16 for `d26`, loaded once, shared by every sequence. **The cache is
per sequence.** At batch 32 and full context the GQA cache is 1.63 GiB against
933 MiB of weights; the 20-KV-head cache would be 8.13 GiB, several times the
model itself, and it, not the parameters, would decide how many users fit on
the card.

It is also a *speed* argument. Decoding one token is memory-bandwidth-bound, not
compute-bound: the arithmetic for a single position is trivial, but every byte of
the cache must be streamed from HBM to compute the attention scores. A 5× smaller
cache means 5× less traffic per token, on the exact critical path.

### A note on dtype

The formula's last factor is `self.embed.weight.dtype`, not an assumption. **The
52 MiB figure is the bf16 number**, and `tinylm.sample` does not give you bf16:

```python
def load_model(ckpt_path: str, device: str) -> tuple[TinyLM, ModelConfig]:
    ckpt = torch.load(ckpt_path, map_location=device, weights_only=True)
    cfg = ModelConfig(**ckpt["model_cfg"])
    model = TinyLM(cfg).to(device)
    model.load_state_dict(ckpt["model"])
    model.eval()
    return model, cfg
```

`TinyLM(cfg)` constructs in PyTorch's default fp32 and the checkpoint's fp32
master weights load straight into it. So sampling from a checkpoint as shipped
runs the model **and its cache** in fp32: **104 MiB at full context, not 52**,
and 1.82 GiB of weights rather than 933 MiB. That is fine for the interactive,
batch-1 use
`tinylm.sample` is for. If you are serving, cast the model (`model.bfloat16()`)
after loading and the cache follows automatically, because its dtype is read off
the embedding rather than hardcoded.

## Sampling

Everything after the forward is turning one logit vector into one token id. The
knobs apply in a fixed order (**temperature → top-k → top-p → sample**), and
the order is why they compose the way they do.

### Greedy: `temperature=0`

```python
if temperature == 0.0:
    next_tok = logits_last.argmax(-1, keepdim=True)
```

Special-cased first, and it must be: `temperature` is a divisor, so 0.0 would be
a division by zero producing `inf`/`nan` rather than the sharp distribution the
limit implies. `argmax` **is** that limit (as `temperature → 0` the softmax
concentrates entirely on the highest logit), so the branch is mathematically the
right answer, not a fallback.

Greedy decoding is deterministic: same prompt, same checkpoint, same output,
every time, no seed involved. **`top_k` and `top_p` are ignored entirely** on
this path (an argmax cannot be affected by truncating the tail below it). Use it
when you want reproducibility or are measuring the model rather than entertaining
a reader.

### Temperature

```python
logits_last = logits_last / temperature
```

Dividing the logits before the softmax rescales the distribution's sharpness:

- **`temperature < 1`**: logits spread apart, softmax sharpens, probability
  concentrates on the top tokens. More conservative, more repetitive.
- **`temperature = 1`**: the model's own distribution, untouched.
- **`temperature > 1`**: logits compress toward each other, the distribution
  flattens, unlikely tokens get real mass. More diverse, more incoherent.

The default in `tinylm.sample` is **0.8**, mildly sharpened, the usual choice
for a base model, where the raw distribution is diffuse enough that `1.0` wanders.

A consequence of the ordering worth knowing: because dividing by a positive
temperature is **monotonic**, it does not change the logits' *ranking*. So
**top-k selects the same k tokens at any temperature**: temperature only changes
how mass is distributed among them. **Top-p is not invariant**, because it
thresholds cumulative *probability*: raising the temperature flattens the
distribution, so more tokens are needed to reach `p`, and the nucleus grows. The
two knobs interact through temperature in genuinely different ways.

### Top-k

```python
if top_k is not None:
    kth = torch.topk(logits_last, min(top_k, logits_last.size(-1))).values[:, -1:]
    logits_last = logits_last.masked_fill(logits_last < kth, float("-inf"))
```

Keep the `k` highest-probability tokens, kill the rest. `torch.topk(...).values[:,
-1:]` is the **k-th largest logit** (the threshold), and everything strictly
below it is set to `-inf`, which softmax maps to exactly zero probability. The
surviving mass is renormalized automatically by the softmax that follows.

The `min(top_k, logits_last.size(-1))` guard keeps `topk` from raising when `k`
exceeds the vocabulary. And note the comparison is `<`, strictly: tokens *tied*
with the k-th logit all survive, so an exact tie can leave more than `k`
candidates. Harmless with real-valued logits, and correct: arbitrarily breaking
a tie would be worse.

Top-k's weakness is that `k` is fixed while the model's confidence is not. At a
position where the model is certain, `k = 50` drags in 49 tokens it had
effectively ruled out. At a genuinely ambiguous position, 50 may cut off
reasonable continuations. The default is **50**.

### Top-p (nucleus)

```python
if top_p is not None:
    sorted_logits, sorted_idx = torch.sort(logits_last, descending=True)
    probs = F.softmax(sorted_logits, dim=-1)
    mask = probs.cumsum(-1) - probs > top_p
    sorted_logits = sorted_logits.masked_fill(mask, float("-inf"))
    logits_last = torch.full_like(logits_last, float("-inf")).scatter(
        1, sorted_idx, sorted_logits
    )
```

Top-p fixes exactly that weakness: instead of a fixed count, keep the smallest set
of tokens whose cumulative probability reaches `p`. The nucleus **adapts**: a
confident position keeps 2 tokens, an ambiguous one keeps 200.

The subtle line is `probs.cumsum(-1) - probs > top_p`. Subtracting `probs` from
its own cumulative sum yields the **exclusive** prefix sum: the mass strictly
*before* each token. So a token is dropped only when the mass ahead of it already
exceeds `p`, which means **the token that crosses the threshold is kept, and the
top-1 token is always kept** (its exclusive prefix is 0, and `0 > p` is false for
any `p > 0`). An inclusive `cumsum(-1) > top_p` would drop the crossing token,
and with a `p` smaller than the top token's own probability it would mask
*everything*: `-inf` across the vocab, `nan` out of the softmax, and
`multinomial` throwing. This one-term shift is the difference between a working
sampler and a crash on `top_p=0.5` at a confident position.

The `scatter` puts the masked logits back in vocabulary order, since the mask was
computed in sorted order and `multinomial` needs vocab-indexed input.

**Top-p runs after top-k**, on already-truncated logits, so its softmax is over
the surviving `k` and its nucleus is a subset of them. Passing both means "at most
`k`, and within that, the `p`-nucleus": the intersection, and the stricter of the
two wins. `tinylm.sample` defaults `top_p` to `None` (off), `top_k` to 50.

### The draw, and stopping

```python
next_tok = torch.multinomial(F.softmax(logits_last, dim=-1), num_samples=1)
next_tok = torch.where(finished[:, None], torch.full_like(next_tok, eos_id), next_tok)
finished |= next_tok.squeeze(1).eq(eos_id)
idx = torch.cat([idx, next_tok], dim=1)
if eos_id is not None and finished.all():
    break
if token_idx + 1 < max_new_tokens:
    logits, _ = self(next_tok, cache=cache)
```

One softmax over the surviving logits, one categorical draw. Note the order at the
bottom: the token is **appended before the break check**, so an emitted `eos` is
included in the returned sequence. A per-row `finished` mask forces subsequent
positions for completed rows to stay at `eos_id` while other rows continue. The
batch stops once every row is finished. The iteration bound also skips the
otherwise-unused decode after the final requested token.

`generate` returns the **full** `idx`: prompt tokens followed by generated ones.
Slice off the prompt if you only want the continuation.

## The CLI

```bash
uv run python -m tinylm.sample \
    --ckpt out/smoke/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json \
    --prompt "The" \
    --max-new-tokens 60 \
    --temperature 0.8 \
    --top-k 50 \
    --seed 1
```

| Flag | Default | Meaning |
|---|---|---|
| `--ckpt` | *required* | Checkpoint path |
| `--tokenizer` | *required* | `tokenizer.json`; **must be the one the model trained on** |
| `--prompt` | `""` | Empty prompt = unconditional generation from `<\|endoftext\|>` alone |
| `--max-new-tokens` | `200` | Tokens to generate |
| `--temperature` | `0.8` | `0` = greedy |
| `--top-k` | `50` | `None` to disable |
| `--top-p` | `None` | Off by default |
| `--seed` | `0` | Seeds `torch.manual_seed`; irrelevant at `--temperature 0` |
| `--device` | auto | `cuda` → `mps` → `cpu` |

Two details in `main` are worth reading:

```python
ids = [tok.eot_id] + tok.encode(args.prompt)
idx = torch.tensor([ids], device=device)
out = model.generate(idx, args.max_new_tokens, temperature=args.temperature,
                     top_k=args.top_k, top_p=args.top_p, eos_id=tok.eot_id)
print(tok.decode(out[0].tolist()[1:]))
```

**The prompt is prefixed with `eot_id`**, and the output is decoded from `[1:]` to
strip it. This is not ceremony. `<|endoftext|>` is the document separator the
model saw between every pair of documents during training
([prepare_data](/docs/data) appends it to each one), so it is the token that means
"a document starts here." Beginning without it asks the model to continue a
document whose start it never saw. It also makes an **empty prompt** well-defined:
`idx` is `[[eot_id]]`, a legal length-1 sequence, and the model generates a
document from nothing, which is exactly how the training loop's periodic samples
work:

```python
prompt = torch.tensor([[tokenizer.eot_id]], device=device)
out = raw_model.generate(prompt, max_new_tokens=100, temperature=0.8, top_k=50)
print("sample:", tokenizer.decode(out[0].tolist()[1:]))
```

Note `eos_id=tok.eot_id` is passed too: the same token that starts a document
ends one, so generation stops naturally at a document boundary rather than always
running the full budget.

For what the output actually looks like at 13M parameters, see the
[recorded generations](/docs/runbooks/local-smoke#sample-generations).
