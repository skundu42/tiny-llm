Two frozen-in-shape dataclasses and two preset dictionaries. `tinyllm/config.py`
is 78 lines and holds every tunable number in the project — nothing is
hardcoded downstream.

## `ModelConfig`

```python
@dataclass
class ModelConfig:
    vocab_size: int = 32768
    n_layer: int = 26
    n_head: int = 20
    n_kv_head: int = 4
    d_model: int = 1280
    d_ff: int = 3456
    seq_len: int = 2048
    rope_theta: float = 10000.0
    norm_eps: float = 1e-6
```

Architecture. The defaults **are** the `d26` preset — 489,297,408 parameters.

| Field | Default | Meaning |
|---|---|---|
| `vocab_size` | `32768` | Tokenizer vocabulary. Must match the tokenizer the model trains on. |
| `n_layer` | `26` | Transformer blocks. |
| `n_head` | `20` | Query heads. |
| `n_kv_head` | `4` | Key/value heads — [GQA](/docs/architecture/attention#gqa-why-4-kv-heads). Must divide `n_head`. |
| `d_model` | `1280` | Residual-stream width. Must divide evenly by `n_head`. |
| `d_ff` | `3456` | [SwiGLU](/docs/architecture/swiglu) inner width (≈ `8/3 × d_model`, rounded to a multiple of 128). |
| `seq_len` | `2048` | Context length. Sizes the RoPE tables and caps generation. |
| `rope_theta` | `10000.0` | [RoPE](/docs/architecture/rope) base frequency. |
| `norm_eps` | `1e-6` | [RMSNorm](/docs/architecture/rmsnorm) epsilon. |

### `__post_init__`

```python
def __post_init__(self) -> None:
    assert self.d_model % self.n_head == 0, "d_model must divide evenly into heads"
    assert self.n_head % self.n_kv_head == 0, "n_head must be a multiple of n_kv_head"
```

**Returns:** `None`. Raises `AssertionError` on an inconsistent config.

Both invariants are structural. `d_model % n_head == 0` makes `head_dim` an
integer; `n_head % n_kv_head == 0` makes the GQA groups equal-sized, so
`repeat_interleave` lines each KV head up with exactly `n_head // n_kv_head` query
heads. Dataclasses call this automatically on construction — an invalid config
fails at the line that creates it, not 200 lines later inside a matmul.

### `head_dim`

```python
@property
def head_dim(self) -> int:
    return self.d_model // self.n_head
```

**Returns:** `int` — `d_model // n_head`; `1280 // 20 = 64` for `d26`.

Derived, not stored, so it cannot drift out of sync with the fields it comes from.

```python
from tinyllm import ModelConfig

cfg = ModelConfig()
assert cfg.head_dim == 64

small = ModelConfig(n_layer=6, d_model=256, n_head=4, n_kv_head=2)
assert small.head_dim == 64

ModelConfig(d_model=100, n_head=8)  # AssertionError: d_model must divide evenly
```

## `TrainConfig`

```python
@dataclass
class TrainConfig:
    # data
    data_dir: str = "data/fineweb-edu"
    # batch: tokens per optimizer step, summed across all ranks
    batch_tokens: int = 524_288
    micro_batch_size: int = 8           # sequences per micro-step per rank
    # schedule
    total_steps: int = 19_073           # ~10B tokens / 524288
    warmup_steps: int = 250
    decay_frac: float = 0.2             # final fraction of steps: linear decay to 0
    # optimizer
    muon_lr: float = 0.02
    muon_momentum: float = 0.95
    adamw_lr: float = 6e-4
    adamw_betas: tuple[float, float] = (0.9, 0.95)
    adamw_wd: float = 0.1               # embeddings only; norm gains get 0
    grad_clip: float = 1.0
    # runtime
    dtype: str = "auto"                 # auto | bf16 | fp32  (auto: bf16 on cuda, fp32 elsewhere)
    compile: bool = True                # auto-disabled off-cuda with a warning
    seed: int = 1337
    # logging / eval / checkpoints
    out_dir: str = "out/d26"
    log_every: int = 10
    val_every: int = 250
    val_batches: int = 20
    ckpt_every: int = 1000
    sample_every: int = 1000
    wandb_project: str = ""             # empty = wandb disabled
```

Everything about the run. No `__post_init__` — validation happens in
[`run`](/docs/api/train#run), which is where `world_size` is known.

### Data and batch

| Field | Default | Meaning |
|---|---|---|
| `data_dir` | `"data/fineweb-edu"` | Directory holding `index.json` and the `.bin` shards. |
| `batch_tokens` | `524_288` | **Tokens per optimizer step, summed across all ranks.** The invariant. |
| `micro_batch_size` | `8` | Sequences per micro-step **per rank** — the memory knob. |

`grad_accum` is **not a field**. It is derived in `run` as `batch_tokens / (world ×
micro_batch_size × seq_len)`, so the effective batch stays fixed as you change GPU
count. See [the grad-accumulation
identity](/docs/training#batch-size-and-the-grad-accumulation-identity).

### Schedule

| Field | Default | Meaning |
|---|---|---|
| `total_steps` | `19_073` | Optimizer steps. `19,073 × 524,288 ≈ 10B` tokens. |
| `warmup_steps` | `250` | Linear LR ramp from `1/warmup` to `1.0`. |
| `decay_frac` | `0.2` | Final fraction of steps that linearly decay to ~0. |

### Optimizer

| Field | Default | Meaning |
|---|---|---|
| `muon_lr` | `0.02` | [Muon](/docs/training/muon) LR — hidden 2-D matrices. |
| `muon_momentum` | `0.95` | Muon's gradient EMA coefficient. |
| `adamw_lr` | `6e-4` | AdamW LR — embedding and norm gains. |
| `adamw_betas` | `(0.9, 0.95)` | AdamW moment decays. |
| `adamw_wd` | `0.1` | Weight decay, **embeddings only** — norm gains get `0.0`. |
| `grad_clip` | `1.0` | Global gradient-norm clip. |

Both LRs are bases: the [WSD schedule](/docs/training#the-wsd-learning-rate-schedule)
multiplies each by the same `lr_scale(step)`.

### Runtime

| Field | Default | Meaning |
|---|---|---|
| `dtype` | `"auto"` | `auto` \| `bf16` \| `fp32`. `auto` = bf16 on CUDA, fp32 elsewhere. |
| `compile` | `True` | `torch.compile`. **Auto-disabled off CUDA** with a printed warning. |
| `seed` | `1337` | Seeds `torch.manual_seed(seed + rank)` and every data draw. |

### Logging, eval, checkpoints

| Field | Default | Meaning |
|---|---|---|
| `out_dir` | `"out/d26"` | Holds `log.csv` and `ckpt_last.pt`. |
| `log_every` | `10` | Steps between train-loss log lines. |
| `val_every` | `250` | Steps between validations — fires on `(step + 1) % val_every == 0`. |
| `val_batches` | `20` | Micro-batches per validation. `20 × 8 × 2048 = 327,680` tokens. |
| `ckpt_every` | `1000` | Steps between checkpoint writes. |
| `sample_every` | `1000` | Steps between sample generations — **only if `--tokenizer` was passed**. |
| `wandb_project` | `""` | W&B project. **Empty disables W&B entirely.** |

## `MODEL_PRESETS`

```python
MODEL_PRESETS: dict[str, ModelConfig] = {
    "d26": ModelConfig(),
    "smoke": ModelConfig(n_layer=6, n_head=4, n_kv_head=2, d_model=256, d_ff=768, seq_len=512),
}
```

| Preset | Params | Shape |
|---|---|---|
| `d26` | **489,297,408** | 26 layers, `d_model` 1280, 20 heads / 4 KV, `d_ff` 3456, ctx 2048 |
| `smoke` | **13,111,296** | 6 layers, `d_model` 256, 4 heads / 2 KV, `d_ff` 768, ctx 512 |

Both counts are asserted **exactly** in the test suite. `smoke` keeps
`vocab_size = 32768`, so it uses the same committed tokenizer and its embedding
(`32768 × 256 = 8,388,608`) is **64% of its parameters** — which is why a 6-layer
model is 13M rather than 5M.

## `TRAIN_PRESETS`

```python
TRAIN_PRESETS: dict[str, TrainConfig] = {
    "d26": TrainConfig(),
    "smoke": TrainConfig(
        data_dir="data/smoke",
        batch_tokens=16_384,
        micro_batch_size=8,
        total_steps=600,
        warmup_steps=20,
        compile=False,
        out_dir="out/smoke",
        val_every=100,
        val_batches=8,
        ckpt_every=200,
        sample_every=200,
    ),
}
```

| | `d26` | `smoke` |
|---|---|---|
| `data_dir` | `data/fineweb-edu` | `data/smoke` |
| `batch_tokens` | 524,288 | 16,384 |
| `micro_batch_size` | 8 | 8 |
| `total_steps` | 19,073 | 600 |
| `warmup_steps` | 250 | 20 |
| `compile` | `True` | `False` |
| `out_dir` | `out/d26` | `out/smoke` |
| `val_every` / `val_batches` | 250 / 20 | 100 / 8 |
| `ckpt_every` / `sample_every` | 1000 / 1000 | 200 / 200 |

Everything else — every optimizer hyperparameter, `grad_clip`, `dtype`, `seed`,
`decay_frac` — is **identical between the two**. That is deliberate: the
[smoke run](/docs/runbooks/local-smoke) exercises the same optimizer split, the
same schedule shape, and the same precision policy as the real run.

<Callout type="warn">
  **The presets are shared mutable instances, not factories.** `MODEL_PRESETS["d26"]`
  returns the *same* `ModelConfig` object every time, and `tinyllm.train`'s
  `main()` mutates it in place from CLI flags (`tc.total_steps = args.steps`).
  Inside a script or notebook that reads a preset twice, use
  `dataclasses.replace(TRAIN_PRESETS["d26"], total_steps=100)` to get an
  independent copy.
</Callout>

## Usage

```python
from dataclasses import replace

from tinyllm import MODEL_PRESETS, TRAIN_PRESETS, ModelConfig, TrainConfig

mc = MODEL_PRESETS["d26"]
print(mc.n_layer, mc.head_dim)          # 26 64

# a copy with an override — does not touch the preset
short = replace(TRAIN_PRESETS["d26"], total_steps=100, out_dir="out/probe")

# an entirely custom pair
mc = ModelConfig(n_layer=12, d_model=768, n_head=12, n_kv_head=4, d_ff=2048)
tc = TrainConfig(batch_tokens=131_072, micro_batch_size=8, total_steps=5_000)
```

Both configs are stored in every checkpoint via `dataclasses.asdict`, which is how
[`load_model`](/docs/api/sample-and-eval#load_model) rebuilds the architecture
without being told what it was.
