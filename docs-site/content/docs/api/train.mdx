## `PEAK_FLOPS`

```python
PEAK_FLOPS = {"NVIDIA H100": 989e12, "NVIDIA A100": 312e12}
```

**Type:** `dict[str, float]`. Dense bf16 peak FLOP/s for each recognized GPU.
`_peak_flops(device)` looks up the current GPU in this dict by matching
`torch.cuda.get_device_name()` against each key **by substring**: a real
device name like `"NVIDIA H100 80GB HBM3"` contains the key `"NVIDIA H100"`.
Off CUDA, or on a GPU that matches no key, it returns `None` and MFU is left
uncomputable for that run rather than guessed.

The master-rank logging branch uses it to compute MFU inline, every
`log_every` steps:

```python
flops_per_token = 6 * raw_model.num_params() + 12 * mc.n_layer * mc.d_model * mc.seq_len
mfu = flops_per_token * tok_s / peak_flops if peak_flops else None
```

The total parameter count is used because the tied embedding matrix also serves
as the dense output projection and therefore performs a matmul for every token.
At `d26`, `flops_per_token ≈ 3.75e9`; at a hypothetical 139k tok/s on an H100
that is `3.75e9 × 139e3 / 989e12` ≈ **53%** MFU, an illustrative number, not a
measured one, since this
repo has not yet run the full `d26` config on real hardware. See
[MFU](/docs/training#logging-throughput-and-mfu).

## `lr_scale`

```python
def lr_scale(step: int, total: int, warmup: int, decay_frac: float) -> float:
    """Warmup-stable-decay multiplier in (0, 1]."""
    if step < warmup:
        return (step + 1) / warmup
    decay_start = int(total * (1 - decay_frac))
    if step < decay_start:
        return 1.0
    return max(1e-8, (total - step) / (total - decay_start))
```

**Args:** `step`: current step (0-based); `total`: total steps; `warmup`:
warmup steps; `decay_frac`: final fraction of steps spent decaying.

**Returns:** `float` in `(0, 1]`: a **multiplier**, not a learning rate. The loop
applies it as `group["lr"] = group["initial_lr"] * scale` to every group of both
optimizers, so Muon's `0.02` and AdamW's `6e-4` share one curve.

A pure function of its arguments: no internal state, nothing to checkpoint,
which is part of why resume is exact.

| Phase | Condition | Value |
|---|---|---|
| Warmup | `step < warmup` | `(step + 1) / warmup`: linear `1/warmup` → `1.0` |
| Stable | `step < int(total × (1 - decay_frac))` | `1.0` |
| Decay | otherwise | linear → `0`, floored at `1e-8` |

```python
from tinylm.train import lr_scale

# the smoke schedule: 600 steps, 20 warmup, 20% decay
print(lr_scale(0, 600, 20, 0.2))      # 0.05,   first step
print(lr_scale(19, 600, 20, 0.2))     # 1.0,    warmup complete
print(lr_scale(300, 600, 20, 0.2))    # 1.0,    plateau
print(lr_scale(500, 600, 20, 0.2))    # 0.833,  decaying (starts at step 480)
print(lr_scale(599, 600, 20, 0.2))    # 0.0083, last step
```

Those are the exact values the [recorded smoke run](/docs/runbooks/local-smoke#what-the-numbers-say)
logged.

## `run`

```python
def run(
    model_cfg: ModelConfig,
    train_cfg: TrainConfig,
    tokenizer_path: str | None = None,
    resume: bool = False,
    device: str | None = None,
) -> dict:
    ...
```

The whole pretraining loop. Everything `main()` does is parse flags and call this.

**Args:**

| Arg | Default | Meaning |
|---|---|---|
| `model_cfg` | | A `ModelConfig` |
| `train_cfg` | | A `TrainConfig` |
| `tokenizer_path` | `None` | If given, loads a tokenizer and prints a sample every `sample_every` steps |
| `resume` | `False` | Load `{out_dir}/ckpt_last.pt` and continue from its step |
| `device` | `None` | Auto-selects `cuda` → `mps` → `cpu`. **Ignored under DDP** |

**Returns:** a `dict`:

```python
{"history": history, "final_val_loss": final_val, "step": tc.total_steps}
```

| Key | Type | Meaning |
|---|---|---|
| `history` | `list[tuple[int, float]]` | `(step, train_loss)` at each `log_every`; **master rank only**, empty elsewhere |
| `final_val_loss` | `float` | Last validation loss; `nan` if none ran |
| `step` | `int` | `train_cfg.total_steps`, the configured total, **not** steps executed |

**Raises:** `ValueError` for invalid config/batch math or an incompatible resume
checkpoint; `FileExistsError` when a fresh run would reuse an existing log or
checkpoint; `FileNotFoundError` if `resume=True` and no checkpoint exists.

**Side effects:** creates `out_dir`; appends to `out_dir/log.csv`; writes
`out_dir/ckpt_last.pt` every `ckpt_every` steps and on the final step; optionally
initializes W&B; under DDP, calls `init_process_group` and `destroy_process_group`.

Behavior worth knowing as a caller:

- **DDP activates on the `RANK` environment variable**: `ddp = "RANK" in
  os.environ`, which `torchrun` sets. There is no flag. Single-GPU is the
  degenerate `world = 1` case of the same code path.
- **`grad_accum` is derived**, never configured: `batch_tokens // (world ×
  micro_batch_size × seq_len)`.
- **`torch.compile` is silently conditional**: `tc.compile and
  device.startswith("cuda")`. Off CUDA it prints `torch.compile disabled: not on
  CUDA` and continues.
- **The seed is per-rank** (`torch.manual_seed(tc.seed + rank)`), while data draws
  are seeded by `(seed, rank, step, micro)` so ranks make independent,
  deterministic draws. Sampling is with replacement, so overlap is possible.
- **Existing output is explicit.** A resumed run appends to its CSV. A fresh run
  refuses an `out_dir` that already contains `log.csv` or `ckpt_last.pt`, preventing
  duplicate step ranges and accidental checkpoint replacement.

```python
from dataclasses import replace

from tinylm import MODEL_PRESETS, TRAIN_PRESETS
from tinylm.train import run

mc = MODEL_PRESETS["smoke"]
tc = replace(TRAIN_PRESETS["smoke"], total_steps=50, out_dir="out/probe")

result = run(mc, tc, tokenizer_path="tokenizer/tokenizer.json", device="cpu")
print(result["final_val_loss"])
print(result["history"][:3])       # [(0, 10.4399), (10, ...), (20, ...)]
```

## `main`

```python
def main() -> None:
    ...
```

The `python -m tinylm.train` entry point. Parses flags, copies the chosen presets
with `dataclasses.replace`, applies overrides to the copies, and calls `run`.

```bash
uv run python -m tinylm.train --config d26 --tokenizer tokenizer/tokenizer.json
```

| Flag | Type | Effect |
|---|---|---|
| `--config` | `{d26, smoke}` | Preset for **both** model and train config. Default **`smoke`** |
| `--data-dir` | `str` | Override `data_dir` |
| `--out-dir` | `str` | Override `out_dir` |
| `--steps` | `int` | Override `total_steps` |
| `--device` | `str` | Override auto-selection |
| `--tokenizer` | `str` | Enable periodic samples |
| `--resume` | flag | Continue from `{out_dir}/ckpt_last.pt` |
| `--no-compile` | flag | Force `compile = False` |
| `--dtype` | `{auto, bf16, fp32}` | Override precision |
| `--wandb` | `str` | Sets `wandb_project` (empty = disabled) |

Two details:

- **`--config` defaults to `smoke`**, not `d26`. Running `python -m tinylm.train`
  bare trains the 13M model.
- **Overrides never mutate module-level presets.** Repeated calls in one process
  start from clean defaults.
- **`--steps 0` is parsed, then rejected by config validation** instead of being
  silently ignored.

```bash
# resume after a preemption
uv run python -m tinylm.train --config d26 \
    --tokenizer tokenizer/tokenizer.json --resume

# 8 GPUs: identical math, ~1/8 the wall clock
uv run torchrun --standalone --nproc_per_node=8 -m tinylm.train \
    --config d26 --tokenizer tokenizer/tokenizer.json

# short CPU probe
uv run python -m tinylm.train --config smoke --device cpu --steps 20
```

## Outputs

### `log.csv`

Written to `{out_dir}/log.csv` by the master rank, flushed on every write. The
header is now `step,split,loss,lr_scale,tok_per_sec,mfu`; the `mfu` column was
added alongside the inline MFU computation above, written as an empty string
whenever it is not computable (anything off CUDA, or a GPU not in
`PEAK_FLOPS`). The real first rows of the [recorded smoke
run](/docs/runbooks/local-smoke)'s `out/smoke/log.csv` predate that column:
they were captured against the old 5-column header, on MPS where `mfu` would
have been empty regardless:

```
step,split,loss,lr_scale,tok_per_sec
0,train,10.439875,0.0500,61272
10,train,9.776255,0.5500,13118
20,train,8.829264,1.0000,13077
...
99,val,6.727361,,
```

`val` rows leave `lr_scale`, `tok_per_sec`, and `mfu` empty. Losses are written
at `.6f` here while stdout prints `.4f`, so the CSV is the more precise record
of the same number.

<Callout type="info">
  The recorded `61272` first row above came from the older timing implementation
  and is inflated. Current runs track the exact number of optimizer steps and
  accumulate training-only wall time, synchronizing before validation,
  checkpoint, sample, and log boundaries. Step 0 and the first interval after a
  resume therefore report one step's real throughput, while eval/checkpoint time
  no longer depresses the following row.
</Callout>

### `ckpt_last.pt`

Written atomically (`tmp` + `os.replace`): a crash mid-write leaves the previous
checkpoint intact:

```python
torch.save(
    {
        "model": raw_model.state_dict(),
        "optimizers": [o.state_dict() for o in optimizers],
        "step": step,
        "model_cfg": asdict(model_cfg),
        "train_cfg": asdict(train_cfg),
    },
    tmp,
)
os.replace(tmp, path)
```

`optimizers` is what makes resume seamless: Muon's momentum buffers and AdamW's
moments ride along, so the loss continues rather than spiking. `model_cfg` is what
lets [`load_model`](/docs/api/sample-and-eval#load_model) rebuild the architecture
without being told it. Note `"step"` is saved as `step + 1`, the **next** step to
run, so a resume does not redo the last one. ~3.8 GiB for `d26`. See
[Checkpointing](/docs/training/checkpointing).
