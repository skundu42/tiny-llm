The two things you do with a finished checkpoint: read what it writes, and measure
what it knows. `tinyllm/sample.py` is 49 lines, `tinyllm/eval_hellaswag.py` is 68.
[Inference](/docs/inference) and [Evaluation](/docs/evaluation) cover the concepts;
this page is the signature reference.

## `tinyllm.sample`

### `load_model`

```python
def load_model(ckpt_path: str, device: str) -> tuple[TinyLLM, ModelConfig]:
    ckpt = torch.load(ckpt_path, map_location=device, weights_only=False)
    cfg = ModelConfig(**ckpt["model_cfg"])
    model = TinyLLM(cfg).to(device)
    model.load_state_dict(ckpt["model"])
    model.eval()
    return model, cfg
```

**Args:** `ckpt_path` — path to a `ckpt_last.pt`; `device` — where to map it.

**Returns:** `(model, cfg)` — a `TinyLLM` in `eval()` mode, and the `ModelConfig`
it was built from.

**The architecture is reconstructed from the checkpoint**, not passed in.
`train.py` stores `asdict(model_cfg)`, so `ModelConfig(**ckpt["model_cfg"])`
rebuilds the exact shape the weights expect — you never have to remember which
preset produced a file, and a mismatch is impossible rather than merely unlikely.
`cfg` is returned because callers need it: `eval_hellaswag` uses `cfg.seq_len` to
decide which examples fit.

Two things to note:

- **`weights_only=False`** is required because the checkpoint holds config dicts
  and optimizer state, not just tensors. **Only load checkpoints you trust** —
  this path can execute arbitrary code via pickle.
- **The model loads in fp32.** `TinyLLM(cfg)` constructs at PyTorch's default
  dtype and the checkpoint's fp32 master weights load straight in. Cast with
  `model.bfloat16()` if you want bf16 — the [KV cache follows
  automatically](/docs/inference#a-note-on-dtype), since its dtype is read off
  `embed.weight`.

```python
import torch

from tinyllm import BPETokenizer
from tinyllm.sample import load_model

model, cfg = load_model("out/smoke/ckpt_last.pt", "cpu")
print(cfg.n_layer, cfg.seq_len, f"{model.num_params():,}")   # 6 512 13,111,296

tok = BPETokenizer.load("tokenizer/tokenizer.json")
ids = torch.tensor([[tok.eot_id] + tok.encode("The")])
out = model.generate(ids, 60, temperature=0.8, top_k=50, eos_id=tok.eot_id)
print(tok.decode(out[0].tolist()[1:]))
```

The `python -m tinyllm.sample` entry point.

```bash
uv run python -m tinyllm.sample --ckpt out/smoke/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json --prompt "The" --max-new-tokens 60
```

| Flag | Default | Meaning |
|---|---|---|
| `--ckpt` | **required** | Checkpoint path |
| `--tokenizer` | **required** | Must be the tokenizer the model trained on |
| `--prompt` | `""` | Empty = unconditional generation |
| `--max-new-tokens` | `200` | Tokens to generate |
| `--temperature` | `0.8` | `0` = greedy |
| `--top-k` | `50` | Top-k truncation |
| `--top-p` | `None` | Nucleus — off by default |
| `--seed` | `0` | `torch.manual_seed`; irrelevant when `--temperature 0` |
| `--device` | auto | `cuda` → `mps` → `cpu` |

The body is five lines, and two of them matter:

```python
ids = [tok.eot_id] + tok.encode(args.prompt)
idx = torch.tensor([ids], device=device)
out = model.generate(idx, args.max_new_tokens, temperature=args.temperature,
                     top_k=args.top_k, top_p=args.top_p, eos_id=tok.eot_id)
print(tok.decode(out[0].tolist()[1:]))
```

**The prompt is prefixed with `eot_id` and the output decoded from `[1:]` to strip
it.** `<|endoftext|>` is the document separator the model saw between every pair of
training documents, so it is the token meaning "a document starts here" — and it
makes an empty prompt well-defined as the length-1 sequence `[[eot_id]]`. It is
also passed as `eos_id`, so generation stops at a document boundary instead of
always running the full budget.

<Callout type="warn">
  **The tokenizer must be the one the model trained on.** Token ids are meaningless
  across tokenizers — a mismatch produces fluent-looking garbage from `sample` and
  exactly-chance scores from `eval_hellaswag`, with no error either way.
  `tokenizer/tokenizer.json` is committed for precisely this reason.
</Callout>

## `tinyllm.eval_hellaswag`

### `ending_losses`

```python
@torch.no_grad()
def ending_losses(model, ctx_ids, ending_ids_list, device) -> list[tuple[float, float]]:
    """Per ending: (sum, mean) cross-entropy over the ending tokens only."""
```

**Args:** `model` — a `TinyLLM` (anything with `(logits, loss)` from
`model(ids)`); `ctx_ids` — `list[int]`, the shared context; `ending_ids_list` —
`list[list[int]]`, candidate endings; `device`.

**Returns:** `list[tuple[float, float]]` — one `(sum, mean)` per ending, in input
order. `sum` is the ending's total cross-entropy in nats; `mean` is that divided
by its token count.

Runs **one forward per ending**, batch size 1, no KV cache — 4 forwards per
HellaSwag example. The scored span is the **ending tokens only**:
`losses[len(ctx_ids) - 1:]` picks exactly the positions predicting
`ids[C] … ids[C+E-1]`. The shared context contributes nothing, which is what makes
the four scores comparable. See
[the scoring walkthrough](/docs/evaluation#scoring-one-example).

Not HellaSwag-specific — it scores any set of continuations against a shared
prefix:

```python
import torch

from tinyllm import BPETokenizer
from tinyllm.eval_hellaswag import ending_losses
from tinyllm.sample import load_model

model, cfg = load_model("out/smoke/ckpt_last.pt", "cpu")
tok = BPETokenizer.load("tokenizer/tokenizer.json")

ctx = tok.encode("The capital of France is")
endings = [tok.encode(" Paris"), tok.encode(" a large purple fish")]
losses = ending_losses(model, ctx, endings, "cpu")

for (s, m), text in zip(losses, [" Paris", " a large purple fish"]):
    print(f"{text!r}: sum {s:.3f} | mean {m:.3f}")

print("argmin sum :", min(range(2), key=lambda i: losses[i][0]))   # acc
print("argmin mean:", min(range(2), key=lambda i: losses[i][1]))   # acc_norm
```

That snippet is also the clearest way to feel why normalization matters: `" Paris"`
is one or two tokens and `" a large purple fish"` is five, so `sum` is biased
toward the short one before the model has an opinion.

The `python -m tinyllm.eval_hellaswag` entry point.

```bash
uv run python -m tinyllm.eval_hellaswag --ckpt out/d26/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json --limit 1000
```

| Flag | Default | Meaning |
|---|---|---|
| `--ckpt` | **required** | Checkpoint path |
| `--tokenizer` | **required** | The tokenizer the model trained on |
| `--limit` | `1000` | Examples to score — the cost knob (4 forwards each) |
| `--device` | auto | `cuda` → `mps` → `cpu` |

Streams HellaSwag's **validation** split (the test labels are not public), encodes
each ending with a **leading space** (`tok.encode(" " + e)`), skips examples that
would exceed `cfg.seq_len` without counting them, and reports both metrics every
100 examples:

```
100: acc 0.2700 | acc_norm 0.3100
...
final (1000 examples): acc 0.2960 | acc_norm 0.3210
```

**Requires `datasets`** — imported lazily inside `main`, along with `load_model`,
so importing `ending_losses` costs nothing.

`acc` argmins the **summed** loss; `acc_norm` argmins the **mean**. Quote
`acc_norm`. Chance is **0.25**; `d26` should land at **0.30-0.35**. See
[what to expect](/docs/evaluation#what-to-expect-at-this-scale).
