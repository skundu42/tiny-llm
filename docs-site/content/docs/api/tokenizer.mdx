A byte-level BPE tokenizer written from scratch in 248 lines:
`tinyllm/tokenizer.py`. No `tokenizers` dependency on the critical path — the HF
library appears only in `export_fast`, as an accelerator that must prove itself
against this implementation before it is trusted. See
[Tokenizer](/docs/tokenizer) for the design.

## Module constants

### `SPLIT_PATTERN`

```python
SPLIT_PATTERN = (
    r"'(?i:[sdmt]|ll|ve|re)"
    r"|[^\r\n\p{L}\p{N}]?\p{L}+"
    r"|\p{N}{1,3}"
    r"| ?[^\s\p{L}\p{N}]+[\r\n]*"
    r"|\s*[\r\n]"
    r"|\s+(?!\S)"
    r"|\s+"
)
```

**Type:** `str`. The GPT-4-style pre-tokenization pattern, compiled with the
`regex` module (`\p{L}` / `\p{N}` need it; `re` will not do).

The comment above it in source explains the one deliberate deviation:

> GPT-4-style pre-tokenization pattern. Possessive quantifiers are replaced by
> greedy ones so the pattern behaves identically under Python `regex` and the
> Rust engine used by HF `tokenizers` (see export_fast in Task 3).

That substitution is what makes `export_fast` able to agree token-for-token.
Saved into `tokenizer.json` so a loaded tokenizer records the pattern it was
trained with — and `load` checks it against today's `SPLIT_PATTERN`, raising
rather than mis-encoding if they differ (see [`load`](#load) below). See
[Tokenizer training](/docs/tokenizer/training).

### `ENDOFTEXT`

```python
ENDOFTEXT = "<|endoftext|>"
```

**Type:** `str`. The only special token. Id **32767** in the canonical tokenizer —
`256 + len(merges)`, i.e. always last.

## `BPETokenizer`

```python
class BPETokenizer:
    def __init__(
        self,
        merges: list[tuple[int, int]] | None = None,
        special_tokens: dict[str, int] | None = None,
    ) -> None:
        ...
```

**Args:** `merges` — ordered list of `(id_a, id_b)` pairs; **order is rank**, and
rank is what decides which merge applies first. `special_tokens` — name → id.

Both default to empty, so `BPETokenizer()` is a valid byte-level tokenizer with
256 tokens and no merges.

**Attributes:** `merges`, `special_tokens`, `merge_ranks` (`pair → rank`), `vocab`
(`id → bytes`).

Construction calls `_build`, which materializes `vocab` by replaying the merges
from the 256 byte tokens up.

### `vocab_size`

```python
@property
def vocab_size(self) -> int:
    return 256 + len(self.merges) + len(self.special_tokens)
```

**Returns:** `int`. For the canonical tokenizer: `256 + 32511 + 1` = **32,768**.

The three terms are the whole vocabulary design: every byte, every learned merge,
one special token. **Byte coverage is total** — there is no `<unk>` and no input
this tokenizer cannot encode.

### `eot_id`

```python
@property
def eot_id(self) -> int:
    return self.special_tokens[ENDOFTEXT]
```

**Returns:** `int` — **32767** canonically. **Raises:** `KeyError` if the
tokenizer has no `<|endoftext|>`.

### `train`

```python
@classmethod
def train(
    cls,
    texts: Iterable[str],
    vocab_size: int,
    num_proc: int = 1,
    min_word_freq: int = 1,
    verbose: bool = False,
) -> "BPETokenizer":
    ...
```

**Args:**

| Arg | Default | Meaning |
|---|---|---|
| `texts` | — | Iterable of documents. **Streamed** — never fully materialized. |
| `vocab_size` | — | Target size. Merges run = `vocab_size - 256 - 1`. |
| `num_proc` | `1` | Processes for pre-tokenization counting. `> 1` uses a `Pool`. |
| `min_word_freq` | `1` | Drop pre-tokens rarer than this before merging. |
| `verbose` | `False` | Print `merge N/M` every 1000 merges. |

**Returns:** a new `BPETokenizer` with `{ENDOFTEXT: 256 + len(merges)}`.

**Raises:** `AssertionError` if `vocab_size <= 257` ("need room for 256 bytes +
`<|endoftext|>`").

Two properties worth relying on:

- **Deterministic.** Ties are broken explicitly — `max(pair_counts.items(),
  key=lambda kv: (kv[1], (-kv[0][0], -kv[0][1])))` — highest count first, then
  smallest pair ids. Same corpus in, same merges out, regardless of dict ordering.
- **Tractable in pure Python** because it counts *unique* pre-tokens once with a
  frequency weight and maintains an incremental `pair_counts` / `pair_to_words`
  index, updating only the words containing the merged pair rather than rescanning
  the corpus.

```python
from tinyllm import BPETokenizer

tok = BPETokenizer.train(
    ["hello world", "hello there", "world of words"] * 100,
    vocab_size=300,
    num_proc=1,
)
print(tok.vocab_size)     # 300
print(tok.encode("hello world"))
```

### `encode`

```python
def encode(self, text: str) -> list[int]:
    """Encode text to token ids. Special tokens are never produced."""
```

**Args:** `text` — any `str`. **Returns:** `list[int]`.

Pre-tokenizes with `SPLIT_PATTERN`, then greedily applies the lowest-rank merge
within each pre-token until none apply. Per-word results are memoized
(`lru_cache(maxsize=1 << 16)`), which is most of the speed on natural text.

<Callout type="warn">
  **Special tokens are never produced by `encode`.** `"<|endoftext|>"` in the input
  encodes as its literal characters, not as id 32767. This is a security property,
  not an oversight — text from a user cannot inject a document boundary. To emit
  one, prepend `tok.eot_id` yourself, as [`tinyllm.sample`](/docs/api/sample-and-eval#main)
  and [`prepare_data`](/docs/api/data#scriptsprepare_datapy) both do.
</Callout>

### `decode`

```python
def decode(self, ids: Sequence[int]) -> str:
    ...
```

**Args:** `ids` — any sequence of ints. **Returns:** `str`.

Special ids map back to their literal text; everything else concatenates
`vocab[i]` bytes and decodes utf-8 with `errors="replace"`. The `replace` matters:
a token boundary can split a multi-byte codepoint, so decoding a *partial* stream
yields `` — never an exception.

```python
from tinyllm import BPETokenizer

tok = BPETokenizer.load("tokenizer/tokenizer.json")
s = "The history of science shows that 42 ideas can change everything."
ids = tok.encode(s)
assert tok.decode(ids) == s                        # exact round-trip
print(len(s.encode()) / len(ids), "bytes/token")   # 5.00
print(tok.decode([tok.eot_id]))                    # <|endoftext|>
```

### `save`

```python
def save(self, path: str) -> None:
    ...
```

**Args:** `path`. **Returns:** `None`. Writes JSON:

```python
data = {
    "version": 1,
    "pattern": SPLIT_PATTERN,
    "merges": self.merges,
    "special_tokens": self.special_tokens,
}
```

**The merges are the tokenizer.** `vocab` is derived on load, never stored — it
cannot drift out of sync with the merges that generate it. See
[Format and the fast path](/docs/tokenizer/format-and-fast-path).

### `load`

```python
@classmethod
def load(cls, path: str) -> "BPETokenizer":
    ...
```

**Args:** `path`. **Returns:** `BPETokenizer`. **Raises:** `ValueError` if the
file's `"pattern"` is present and does not match today's `SPLIT_PATTERN`.

JSON turns tuples into lists, so merges are re-tupled on the way in
(`[tuple(m) for m in data["merges"]]`) — `merge_ranks` is keyed by tuple and lists
are unhashable.

`"pattern"` **is now validated**, not just recorded: if `data.get("pattern")` is
present and differs from the module-level `SPLIT_PATTERN`, `load` raises rather
than silently reloading under today's pattern. A tokenizer saved under a
different `SPLIT_PATTERN` would otherwise load "successfully" and mis-encode
every input — the merges are byte-pair ids trained against a specific
pre-tokenization, and pre-tokenizing differently changes which words see which
merges. Files with no `"pattern"` key (or `null`) load unchanged, for
compatibility with tokenizers saved before this field existed.

### `export_fast`

```python
def export_fast(self):
    """Build a HF `tokenizers` Tokenizer with identical merges.

    Used only to accelerate bulk corpus encoding; call verify_fast on a
    sample before trusting it. Our pure-Python encoder is ground truth.
    """
```

**Args:** none. **Returns:** a `tokenizers.Tokenizer`. **Imports `tokenizers`
lazily** — the dependency is not needed to train, encode, or decode.

Rebuilds the vocab through GPT-2's reversible byte→printable-unicode table, since
HF's BPE model is keyed by `str` and ours by `bytes`. Roughly **100× faster** for
bulk encoding, which is the only reason it exists.

### `verify_fast`

```python
def verify_fast(self, fast, texts: Iterable[str]) -> None:
    """Assert `fast` produces byte-identical ids to our encoder."""
```

**Args:** `fast` — a tokenizer from `export_fast`; `texts` — documents to check.
**Returns:** `None`. **Raises:** `ValueError` on the first mismatch, quoting the
offending text.

```python
import itertools

from datasets import load_dataset

from tinyllm import BPETokenizer

tok = BPETokenizer.load("tokenizer/tokenizer.json")
ds = load_dataset("HuggingFaceFW/fineweb-edu", name="sample-10BT",
                  split="train", streaming=True)
docs = (ex["text"] for ex in ds)

fast = tok.export_fast()
tok.verify_fast(fast, itertools.islice(docs, 200))   # raises on any disagreement
ids = [e.ids for e in fast.encode_batch(["hello world", "goodbye"])]
```

This is exactly what `prepare_data.py` does before encoding a corpus — 200 real
documents, and the run aborts if a single id differs.

## `scripts/train_tokenizer.py`

```bash
uv run python scripts/train_tokenizer.py --max-bytes 250000000
```

| Flag | Default | Meaning |
|---|---|---|
| `--out` | `tokenizer/tokenizer.json` | Output path |
| `--max-bytes` | `250_000_000` | Bytes of FineWeb-Edu to train on |
| `--vocab-size` | `32768` | Target vocab |
| `--num-proc` | `max(1, os.cpu_count() - 2)` | Counting processes |
| `--min-word-freq` | `2` | Drop pre-tokens seen once |

Streams FineWeb-Edu `sample-10BT`, trains, saves, then measures compression on
**100 held-out documents** — taken from the same iterator *after* training, so
they are genuinely unseen:

```python
holdout = [ex["text"] for ex in itertools.islice(it, 100)]
nbytes = sum(len(t.encode("utf-8")) for t in holdout)
ntok = sum(len(tok.encode(t)) for t in holdout)
print(f"compression on holdout: {nbytes / ntok:.3f} bytes/token")
```

**Measured:** `4.517 bytes/token`. The canonical `tokenizer/tokenizer.json` is
**committed** — see the [smoke runbook](/docs/runbooks/local-smoke#step-2-train-the-canonical-tokenizer)
for why you should not retrain it.
