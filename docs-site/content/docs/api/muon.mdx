92 lines: `tinyllm/muon.py`. An optimizer that orthogonalizes each 2-D weight's
momentum-smoothed gradient with a quintic Newton-Schulz iteration, plus the
function that decides which parameters get it. [Muon](/docs/training/muon) derives
the method; this page is the signature reference.

## `zeropower_via_newtonschulz5`

```python
def zeropower_via_newtonschulz5(G: torch.Tensor, steps: int = 5) -> torch.Tensor:
    """Approximate UV^T for G = USV^T via quintic Newton-Schulz in bfloat16."""
```

**Args:** `G`: a **2-D** tensor; `steps`: iterations (default `5`).

**Returns:** a tensor of `G`'s shape and dtype, approximating `U Váµ€`: `G` with
every singular value driven to â‰ˆ 1.

**Raises:** `ValueError` if `G.ndim != 2`.

Computes the orthogonalization **without ever forming an SVD**. Internals worth
knowing as a caller:

- Runs in **bfloat16** internally and casts back to `G.dtype` on the way out.
- **Normalizes first** (`X / (X.norm() + 1e-7)`) into the basin the fixed
  coefficients `(3.4445, -4.7750, 2.0315)` converge from. Passing an unnormalized
  matrix is fine; it does this itself.
- **Transposes tall inputs** so the Gram matmuls run on the smaller side, and
  transposes back.
- **Pure function.** `G` is not modified in place.

```python
import torch

from tinyllm.muon import zeropower_via_newtonschulz5

G = torch.randn(64, 128)
X = zeropower_via_newtonschulz5(G)
print(X.shape, X.dtype)                     # torch.Size([64, 128]) torch.float32

s = torch.linalg.svdvals(X.float())
print(s.min().item(), s.max().item())       # both â‰ˆ 1.0
```

## `Muon`

```python
class Muon(torch.optim.Optimizer):
    def __init__(self, params, lr: float = 0.02, momentum: float = 0.95,
                 nesterov: bool = True, ns_steps: int = 5) -> None:
        ...
```

**Args:**

| Arg | Default | Meaning |
|---|---|---|
| `params` |  | Iterable of parameters or param groups. **2-D only.** |
| `lr` | `0.02` | Learning rate: ~30Ã— AdamW's, because the update is unit-scaled |
| `momentum` | `0.95` | Gradient EMA coefficient |
| `nesterov` | `True` | Blend the fresh gradient with the buffer before orthogonalizing |
| `ns_steps` | `5` | Newton-Schulz iterations |

A standard `torch.optim.Optimizer`: `state_dict()` / `load_state_dict()` work, and
that is how [checkpointing](/docs/training/checkpointing) preserves the momentum
buffers across a resume.

**State:** one `momentum_buffer` per parameter, **half of AdamW's two moments**,
which is a real memory saving on the 447M parameters it owns here.

<Callout type="warn">
  **Every parameter given to `Muon` must be 2-D.** `zeropower_via_newtonschulz5`
  asserts `G.ndim == 2`, so passing a norm gain or a bias raises. Use
  [`build_optimizers`](#build_optimizers) rather than constructing it by hand.
</Callout>

### `step`

```python
@torch.no_grad()
def step(self) -> None:
    ...
```

**Args:** none; **no `closure` parameter**, unlike most PyTorch optimizers.
**Returns:** `None`.

Per parameter: update the momentum EMA â†’ optionally apply Nesterov â†’ orthogonalize
â†’ scale by `max(1, rows/cols) ** 0.5` â†’ `p -= lr Ã— scale Ã— update`. Parameters with
`grad is None` are skipped.

Note `step` does **not** zero gradients; the training loop calls
`opt.zero_grad(set_to_none=True)` explicitly.

<Callout type="warn">
  **The Nesterov branch mutates `p.grad` in place** (`g = g.lerp_(buf,
  momentum)`, where `g is p.grad`). Source has a comment on this; it is safe
  here for three reasons specific to this training loop: gradient clipping
  runs before `step()`, so nothing reads the pre-mutation gradient afterward;
  Muon and AdamW own disjoint parameter sets, so no other optimizer touches
  this `p.grad`; and `zero_grad(set_to_none=True)` follows immediately,
  discarding the buffer rather than reusing it. **It would break under DDP's
  `gradient_as_bucket_view=True`**, which aliases `p.grad` to a communication
  bucket; mutating it in place would corrupt the bucket instead of just the
  local gradient. `tinyllm`'s DDP wrapper does not set that flag.
</Callout>

## `build_optimizers`

```python
def build_optimizers(model: nn.Module, cfg: TrainConfig) -> list[torch.optim.Optimizer]:
    """Muon for hidden 2-D matrices; AdamW for the (tied) embedding and norm gains."""
```

**Args:** `model`: any `nn.Module`; `cfg`: a `TrainConfig` supplying `muon_lr`,
`muon_momentum`, `adamw_lr`, `adamw_betas`, `adamw_wd`.

**Returns:** `[muon, adamw]`: **two** optimizers, in that order. The training loop
steps both.

The sort is by **shape and name**:

```python
for name, p in model.named_parameters():
    if not p.requires_grad:
        continue
    if p.ndim == 2 and "embed" not in name:
        hidden.append(p)
    elif p.ndim == 2:
        embed.append(p)
    else:
        norms.append(p)
```

| Bucket | Rule | Contents (`d26`) | Optimizer |
|---|---|---|---|
| `hidden` | 2-D, name lacks `"embed"` | 7 matrices Ã— 26 layers: `wq`, `wk`, `wv`, `wo`, `w_gate`, `w_up`, `w_down` (**447,283,200** params) | **Muon**, `lr=0.02` |
| `embed` | 2-D, name has `"embed"` | The tied token embedding (**41,943,040**) | **AdamW**, `lr=6e-4`, `wd=0.1` |
| `norms` | everything else (1-D) | Every `RMSNorm` gain (**71,168**) | **AdamW**, `lr=6e-4`, `wd=0.0` |

**Weight tying is why the name check works.** `lm_head.weight` *is*
`embed.weight`, and `named_parameters()` deduplicates shared storage: the tensor
is yielded once, under the `embed.weight` name. So it lands in the `embed` bucket
and never in `hidden`, despite being the output projection too.

**Every group gets an `initial_lr` snapshot:**

```python
for opt in (muon, adamw):
    for group in opt.param_groups:
        group["initial_lr"] = group["lr"]
```

This is the hook the [WSD schedule](/docs/training#the-wsd-learning-rate-schedule)
scales each step (`group["lr"] = group["initial_lr"] * scale`), so Muon's `0.02`
and AdamW's `6e-4` ride the same curve off their own bases. **A hand-built
optimizer without `initial_lr` will `KeyError` on the first step of `run`.**

```python
from tinyllm import MODEL_PRESETS, TRAIN_PRESETS, TinyLLM
from tinyllm.muon import build_optimizers

model = TinyLLM(MODEL_PRESETS["d26"])
muon, adamw = build_optimizers(model, TRAIN_PRESETS["d26"])

n_muon = sum(p.numel() for g in muon.param_groups for p in g["params"])
n_adamw = sum(p.numel() for g in adamw.param_groups for p in g["params"])
print(f"{n_muon:,} / {n_adamw:,}")       # 447,283,200 / 42,014,208
assert n_muon + n_adamw == model.num_params()

# the training loop's per-step LR scaling
for opt in (muon, adamw):
    for group in opt.param_groups:
        group["lr"] = group["initial_lr"] * 0.5
```
