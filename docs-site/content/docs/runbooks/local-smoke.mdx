title: Local smoke run
description: The exact command sequence that trains a real 13.1M model end to end on a laptop, with the measured output of every step from a recorded Apple M3 Pro run
---

This is the runbook that proves the repo works. Every command below was run on an
**Apple M3 Pro (18 GB), PyTorch MPS backend**, against the committed code, and
**every number on this page is a measurement from that run**, not an estimate,
not a projection. Total time is about 15 minutes of training plus a tokenizer and
a data step you can shorten or skip. It assumes you are comfortable with the
tooling; if this is your first time training anything, follow
[Your first training run](/docs/runbooks/first-run), which walks the same
sequence with every step explained.

The `smoke` preset is not a toy reimplementation. It is the same `tinylm.train`
entry point, the same [Muon+AdamW split](/docs/training/muon), the same
[WSD schedule](/docs/training), the same [data pipeline](/docs/data) and the
same [model code](/docs/architecture) as the 489M `d26` run; only the config
differs. If this passes, the pipeline is wired correctly and the only thing
`d26` adds is money.

<Callout type="info">
  No GPU is required and nothing here needs CUDA. `torch.compile` disables itself
  off CUDA (with a printed warning) and precision falls back to fp32, so the
  commands need no extra flags on a Mac.
</Callout>

## Step 0: Install

```bash
git clone <repo-url> tiny-lm
cd tiny-lm
uv sync
```

Python 3.12+ is required (`pyproject.toml` pins `requires-python = ">=3.12"`).

## Step 1: Run the tests

```bash
uv run pytest -q
```

Do this before spending 15 minutes on a run. The suite needs no network and no
downloaded data, and it covers every component against hand-derived references:
SDPA against a [naive masked-softmax
reference](/docs/architecture/attention#the-reference-oracle), RoPE's
norm-preservation and relative-position properties, both presets' parameter counts
asserted **exactly** (489,297,408 and 13,111,296), KV-cache decode against a full
uncached forward, and bit-exact resume.

## Step 2: Train the canonical tokenizer

<Callout type="info">
  **Skippable.** `tokenizer/tokenizer.json` is a **committed artifact**: the
  canonical tokenizer every recorded number on this page and in
  [cloud pretraining](/docs/runbooks/cloud-pretrain) uses. Skip to Step 4 unless
  you specifically want to reproduce it. Training your own produces a *different*
  tokenizer, and a checkpoint is only meaningful with the tokenizer it trained on.
</Callout>

```bash
uv run python scripts/train_tokenizer.py --max-bytes 250000000
```

~30-60 minutes on CPU, streaming ~250 MB of FineWeb-Edu `sample-10BT`. It prints
its configuration, then a `merge N/32511` progress line every 1000 merges:

```
training vocab=32768 on ~250MB, num_proc=9
merge 1000/32511
...
saved tokenizer/tokenizer.json
compression on holdout: 4.517 bytes/token
```

`--num-proc` defaults to `max(1, os.cpu_count() - 2)`: 9 workers on the recorded
machine.

**Measured result:**

| Metric | Value |
|---|---|
| Vocab size | **32,768** (32,511 merges + 256 byte tokens + `<\|endoftext\|>`) |
| Training corpus | ~250 MB of FineWeb-Edu `sample-10BT` (streamed) |
| Workers | 9 processes |
| Compression (100 held-out docs) | **4.517 bytes/token** |
| `eot_id` | 32767 |

4.517 bytes/token on documents the trainer never saw is the number that matters:
it is measured on a holdout, so it reflects generalization rather than
memorization. See [Tokenizer](/docs/tokenizer) for why 32,768 and how the trainer
stays tractable in pure Python.

## Step 3: Tokenizer sanity check

```bash
uv run python -c "
from tinylm.tokenizer import BPETokenizer
t = BPETokenizer.load('tokenizer/tokenizer.json')
assert t.vocab_size == 32768 and t.eot_id == 32767
s = 'The history of science shows that 42 ideas can change everything.'
ids = t.encode(s); assert t.decode(ids) == s
print(len(s.encode()) / len(ids), 'bytes/token on sample sentence')"
```

Three assertions in five lines: the vocab is the size the model config expects,
`eot_id` is where it should be, and **encode/decode round-trips exactly**. The
round-trip is the important one: a byte-level BPE that does not reproduce its
input is silently corrupting your corpus.

**Measured:** `5.00 bytes/token` on that sentence. Higher than the 4.517 holdout
figure because one clean, short sentence of common English is easier to compress
than the average web document.

## Step 4: Prepare the smoke dataset

```bash
uv run python scripts/prepare_data.py --data-dir data/smoke \
    --max-tokens 60000000 --val-tokens 2000000 --shard-tokens 25000000
```

Streams FineWeb-Edu, encodes it, and writes uint16 shards plus `index.json`. It
first checks the fast path against the ground-truth encoder and refuses to
continue if they disagree:

```
fast tokenizer verified token-identical on 200 docs
done: 60,003,098 tokens (2,004,611 val, 57,998,487 train) in data/smoke
```

That verification is not optional and not decorative. The corpus is encoded by a
Rust `tokenizers` BPE built from our merges (~100× faster than the pure-Python
encoder), but **our encoder is the ground truth**: `verify_fast` asserts the two
produce byte-identical ids on 200 real documents before a single token is
written. Details in [Tokenizer format and the fast
path](/docs/tokenizer/format-and-fast-path).

**Measured result:**

| Metric | Value |
|---|---|
| Total tokens | **60,003,098** |
| Validation | 2,004,611 (`val_00000.bin`, `val_00001.bin`) |
| Train | 57,998,487 (`train_00000.bin`…`train_00002.bin`) |
| Shard size | 25M tokens (uint16) |
| Fast tokenizer | verified token-identical on 200 docs before use |

The resulting `data/smoke/index.json`:

```json
{
 "dtype": "uint16",
 "splits": {
  "val": [
   {"file": "val_00000.bin", "tokens": 2000000},
   {"file": "val_00001.bin", "tokens": 4611}
  ],
  "train": [
   {"file": "train_00000.bin", "tokens": 25000000},
   {"file": "train_00001.bin", "tokens": 25000000},
   {"file": "train_00002.bin", "tokens": 7998487}
  ]
 }
}
```

Three things this makes concrete:

- **Counts land slightly above the request**: 60,003,098 against `--max-tokens
  60000000`, and 2,004,611 against `--val-tokens 2000000`. The writer never
  truncates mid-document: it finishes the batch of documents it is in, then
  stops. A ~3k-token overshoot on 60M is the tail of the final document batch.
- **Val is filled first, then train.** The router is
  `w = val_w if val_w.total_written < args.val_tokens else train_w`: every
  document goes to validation until the quota is met, and every one after goes to
  train. The splits are therefore **disjoint by construction**, with no document
  spanning both.
- **Shard sizes follow the flags.** Val shards are capped at `min(shard_tokens,
  val_tokens)` = 2M, so 2,004,611 val tokens become a full 2M shard plus a 4,611
  remainder. Train's 57,998,487 at 25M each becomes 25M + 25M + 7,998,487.

Disk cost is ~120 MB (60M tokens × 2 bytes for uint16).

## Step 5: Train

```bash
uv run python -m tinylm.train --config smoke --device mps \
    --tokenizer tokenizer/tokenizer.json
```

The header line reports the derived configuration:

```
params: 13,111,296 | device mps | world 1 | grad_accum 4 | steps 600
```

Every number there is derived, not configured. **13,111,296 params** is the
`smoke` preset (6 layers, `d_model` 256, 4 heads / 2 KV heads, `d_ff` 768, context
512) and is asserted exactly in the test suite. **`grad_accum 4`** is computed
from the batch identity (`batch_tokens / (world × micro_batch_size × seq_len)` =
`16,384 / (1 × 8 × 512)` = 4) rather than read from config, which is what keeps
the effective batch constant across hardware.

`--tokenizer` is optional; passing it enables the periodic sample prints at
`sample_every = 200`. Progress goes to stdout and to `out/smoke/log.csv`
(`step, split, loss, lr_scale, tok_per_sec, mfu`; `mfu` stays empty off CUDA,
so every row from this MPS run has it blank); checkpoints land in
`out/smoke/ckpt_last.pt` every 200 steps.

### The measured curve

Straight from `out/smoke/log.csv`, the full 600-step schedule:

| Step | Train loss | Val loss |
|---|---|---|
| 0 | **10.4399** |  |
| 99 |  | 6.7274 |
| 100 | 6.6888 |  |
| 199 |  | 6.2567 |
| 200 | 6.1561 |  |
| 299 |  | 5.9739 |
| 300 | 5.8912 |  |
| 399 |  | 5.7768 |
| 400 | 5.6471 |  |
| 499 |  | 5.6177 |
| 500 | 5.5151 |  |
| 599 | 5.5300 | **5.5012** |

Validation fires on `(step + 1) % val_every == 0` with `val_every = 100`, which is
why the val points sit at 99, 199, … and the train points at the `log_every = 10`
multiples. Each val number is a mean over 8 batches × 8 sequences × 512 tokens =
32,768 tokens, drawn from **fixed seeded crops**: the same crops at every
evaluation, which is what makes the column comparable down the page. See
[Evaluation](/docs/evaluation#validation-loss).

### What the numbers say

**The first number is the most informative one in the project.** The initial loss
is **10.4399**, and `ln(32768) = 10.3972`. A model that has learned nothing must
spread probability uniformly over the vocabulary, giving a cross-entropy of
exactly `ln(vocab_size)`. Landing **0.04 nats** above it says the
[initialization](/docs/architecture/embeddings-and-init) is correct: the logits
start near-uniform rather than confidently wrong, which is what the 0.02 init and
the `1/√(2L)` residual scaling are for. Had it started at 15, or at 3, something
would be deeply wrong before a single gradient was taken.

**Val tracks train the whole way down** (6.7274 → 6.2567 → 5.9739 → 5.7768 →
5.6177 → 5.5012, no divergence) and at the end val (5.5012) sits just *below*
train (5.5300). No overfitting, exactly as expected: 600 steps × 16,384 =
**9,830,400 tokens** out of a 57,998,487-token training split is under a fifth of
an epoch, so no example is ever seen twice and train and val crops are equally
unfamiliar to the model.

**Final val loss 5.50**: comfortably below the plan's ≤6.0 acceptance gate, and a
perplexity of ~245 over a 32,768-token vocab.

**Throughput held at ~11,000-13,400 tok/s** on MPS, putting the 600 steps at
**≈15 minutes** of wall clock.

**The LR schedule behaved as designed:** warmup reached `lr× 1.0` by step 20
(`warmup_steps = 20`), held a stable plateau, and decay engaged at step 480
(`int(600 × (1 - 0.2))`), falling through `lr× 0.833` at step 500 to `0.008` at
step 599. That trapezoid is the [WSD schedule](/docs/training#the-wsd-learning-rate-schedule)
doing its job.

## Step 6: Verify resume

```bash
uv run python -m tinylm.train --config smoke --device mps \
    --tokenizer tokenizer/tokenizer.json --steps 620 --resume
```

`--resume` loads `out/smoke/ckpt_last.pt` (model weights **and both optimizers'
state**) and continues from the stored step:

```
resumed from out/smoke/ckpt_last.pt at step 600
```

**Measured:** resumed at step 600, ran 20 more steps, val loss **5.4914** at step
619, continuing the curve smoothly down from 5.5012 rather than jumping. That is
the tell. A resume that drops the optimizer state produces a visible loss spike as
Muon's momentum buffers and AdamW's moments rebuild from zero; this one does not,
because `_save_ckpt` stores `[o.state_dict() for o in optimizers]` alongside the
weights and the training RNG is a pure function of `(seed, rank, step, micro)`
rather than a stateful generator.

This matters far more than it looks. **Checkpoint/resume is verified on a real
run, not just in tests**, and it is the mechanism a
[preempted cloud run](/docs/runbooks/cloud-pretrain#recovering-from-a-preemption)
depends on to not lose 20 hours. See
[Checkpointing](/docs/training/checkpointing) for the atomic-write and
bit-exactness story.

## Step 7: Sample

```bash
uv run python -m tinylm.sample --ckpt out/smoke/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json --prompt "The" --max-new-tokens 60
```

### Sample generations

These are the real recorded outputs.

**At step 400**, from the training loop's periodic sample (temperature 0.8, top-k
50, unconditional):

> You need to know how to make sure that you need to be able to get down the next day!
> A. The following you
> - It's important to do that this is the same. When it's an example of what you are going to a different way.

**At step 600** via `tinylm.sample`, prompt `"The"` (seed 1):

> The 'the last in the first years of the past was the most common in the world. The best way to be the most diverse and highly difficult, while there was a lot of the most important for future lives. In a very high level of work, the world's most important is the same as

**Prompt `"Photosynthesis is"`** (seed 2):

> Photosynthesis is a problem that has been published as a new disease in the past 20 years. There are also some studies, which are called "A disease" and is found on these problems in people by the first 20 years in the United States. The most common number of people is about 1.

### How to read this

At 13M parameters and 600 steps, ~10M tokens seen, the model has learned
**English morphology, syntax, punctuation, and even quotation and list
formatting**. The sentences are grammatical and fluent. Subject-verb agreement
holds. Clauses nest correctly.

It has **not** learned factual grounding, and the third sample is the cleanest
possible demonstration: *"Photosynthesis is a problem that has been published as
a new disease."* Every word is a plausible English continuation of the last few;
none of them are about photosynthesis. The model has learned the *shape* of
language and nothing about the world.

**This is exactly what this scale and token budget predict, and it is the correct
outcome for this run.** The point of the smoke run is that every component works
end to end, not that the result is good. Semantic coherence is what the
[`d26` config](/docs/runbooks/cloud-pretrain) buys with 489M parameters and 10B
tokens; you cannot get it from 13M parameters and 0.1% of the token budget, and a
smoke run that produced it would mean the eval was lying.

## What this run establishes

| Claim | Evidence from this run |
|---|---|
| Initialization is correct | Loss starts at 10.4399 vs `ln(32768)` = 10.3972 |
| The tokenizer round-trips and compresses | 4.517 bytes/token on holdout; encode/decode assert passes |
| The fast path is trustworthy | Token-identical to the reference encoder on 200 docs |
| Data splits are disjoint and complete | 2,004,611 val + 57,998,487 train = 60,003,098 |
| The loop learns | 10.44 → 5.50 val over 600 steps |
| The optimizer split works | Loss falls under Muon on hidden matrices + AdamW on embed/norms |
| The LR schedule works | `lr×` 0.05 → 1.0 by step 20 → 0.008 at 599 |
| Checkpoint/resume is sound | Resumed at 600 → 5.4914 at 619, no spike |
| Generation works | Grammatical English from a 60-token prompt |

Everything above is the same code path `d26` takes. Two directions from here:
[fully local training](/docs/runbooks/local-full-train) takes this exact config
to its full compute-optimal token budget overnight on the same laptop, and the
[cloud pretraining runbook](/docs/runbooks/cloud-pretrain) is the real thing:
the same commands with `--config d26`, on an H100, for 20 hours.
