This is the runbook for the real run: **489,297,408 parameters over ~10B
FineWeb-Edu tokens**, about **20 hours on a single H100**, for roughly **$50-60**.
Everything is the same code path the [smoke run](/docs/runbooks/local-smoke)
already proved on a laptop: `--config d26` swaps the preset and nothing else.

The one thing that distinguishes this from the smoke run is that mistakes now cost
money. The order of operations below is built around that: **verify everything
cheap before you start anything expensive**, and never let a typo run for 20 hours
unattended.

## Why 10B tokens

`d26` trains on ~10B tokens for 489M parameters, a ratio of **20.4 tokens per
parameter**, which is essentially exactly [Chinchilla-optimal](https://arxiv.org/abs/2203.15556)
(Hoffmann et al., 2022, found ~20:1 minimizes loss for a fixed compute budget).
This is not a coincidence and not a knob to fiddle: the config is sized so that
FineWeb-Edu's `sample-10BT` subset, which is exactly ~10B tokens, is consumed
in almost exactly one epoch, so **no document is seen twice**.

The step count falls out of the same arithmetic:

```
total_steps = 19,073
batch_tokens = 524,288          (2^19)
19,073 × 524,288 = 9,999,745,024 ≈ 10B tokens
```

## Picking a provider

Prices below are **indicative on-demand H100 80GB rates at the time of writing and
move constantly**: treat them as a shape, not a quote, and check current pricing
before booking. `scripts/cloud_setup.sh` is written for a fresh Ubuntu GPU box and
targets Lambda/RunPod-style images.

| Provider | ~$/h (1× H100) | Preemptible? | Notes |
|---|---|---|---|
| **Lambda Labs** | ~$2.5-3 | No (on-demand) | Ubuntu + CUDA preinstalled, persistent storage; often capacity-constrained |
| **RunPod** | ~$2-3 (Secure Cloud) | Community Cloud: yes | Cheap Community tier is interruptible; Secure Cloud is not |
| **Vast.ai** | ~$1.5-2.5 | Often | Marketplace: cheapest, most variable; **verify disk and bandwidth** before renting |
| **CoreWeave** | ~$2.5-5 | No | Enterprise-oriented, 8× nodes with fast interconnect |
| **AWS p5 / GCP A3** | ~$4-8 | Spot: yes | Most expensive on-demand; use only if you're already there |

What actually matters for this run, in order:

1. **Disk ≥ 60 GB.** The token shards are **~19 GiB** and the HuggingFace
   `datasets` streaming cache is transient but not free. This is the single most
   common way a run dies at hour 2.
2. **Preemptibility.** Cheap tiers are interruptible. That is *fine here* (the
   run [resumes bit-exactly](#recovering-from-a-preemption)) but only if you
   understand you will lose up to `ckpt_every = 1000` steps of work per
   interruption and that persistent storage survives the eviction.
3. **CPU cores.** [Data prep](#step-3-prepare-the-data) is a **single-threaded
   ~1-3 hour CPU job** on a box you are paying GPU rates for. More cores do not
   help `prepare_data.py` directly, but a starved box makes it worse.
4. **Bandwidth.** You are streaming ~10B tokens' worth of text from
   HuggingFace.

<Callout type="warn">
  You pay GPU rates during the 1-3 hour data-prep step while the GPU sits idle;
  that is ~$5-9 of an ~$55 run spent on a CPU task. If your provider supports
  attaching a volume, prepare the data on a cheap CPU instance once and mount it,
  or snapshot `data/fineweb-edu` after the first run and reuse it.
</Callout>

## Step 1: Provision and clone

Provision an **H100 80GB** with **≥60 GB disk**, then:

```bash
git clone <repo-url> tiny-llm
cd tiny-llm
```

## Step 2: Bootstrap

```bash
bash scripts/cloud_setup.sh
```

The whole script is 20 lines and does exactly two things (install `uv` if it is
missing, and `uv sync`), then prints the run sequence for you:

```bash
command -v uv >/dev/null || curl -LsSf https://astral.sh/uv/install.sh | sh
export PATH="$HOME/.local/bin:$PATH"

uv sync
```

Note the `export PATH` applies **inside the script's shell**. If `uv` was just
installed, add `export PATH="$HOME/.local/bin:$PATH"` to your own shell (or open a
new one) before running the `uv run` commands below.

The script's closing hint is conditional in a way worth noticing:

```bash
if [ ! -f tokenizer/tokenizer.json ]; then
  echo "uv run python scripts/train_tokenizer.py            # ~30-60 min CPU"
fi
```

It only suggests training a tokenizer **if one is missing**, and in a fresh clone
it never is.

<Callout type="info">
  **`tokenizer/tokenizer.json` is committed, so skip the tokenizer step.** It is
  the canonical 32,768-vocab tokenizer (4.517 bytes/token on holdout) that all
  recorded numbers use. Training your own would produce a *different* tokenizer
  for no benefit, cost an hour of GPU-rate time, and make your checkpoint
  incomparable to every number in these docs.
</Callout>

Verify before spending anything:

```bash
uv run pytest -q
```

## Step 3: Prepare the data

```bash
uv run python scripts/prepare_data.py
```

The defaults **are** the `d26` configuration: `--tokenizer
tokenizer/tokenizer.json`, `--data-dir data/fineweb-edu`, `--val-tokens
10_000_000`, `--shard-tokens 100_000_000`, `--max-tokens 0` (meaning the entire
subset). No flags needed.

- **~1-3 hours**, CPU-bound, streaming FineWeb-Edu `sample-10BT`.
- **~19 GiB** on disk (10B tokens × 2 bytes, uint16).
- Prints `fast tokenizer verified token-identical on 200 docs`, then a progress
  line every 100M tokens, then a `done:` summary.

Run it in `tmux` too: a dropped SSH session kills it just as dead as it kills a
training run, and this one has no `--resume`.

## Step 4: The 15-minute dress rehearsal

**Do not skip this.** It costs about **$1** and it is the difference between
finding out your config is wrong now or 20 hours and $60 from now.

```bash
tmux new -s rehearsal
uv run python -m tinyllm.train --config d26 \
    --tokenizer tokenizer/tokenizer.json \
    --out-dir out/rehearsal --steps 200
```

Let it run ~15 minutes, then check **three things**:

| Check | Expected | If it fails |
|---|---|---|
| **Header line** | `params: 489,297,408 \| device cuda \| world 1 \| grad_accum 32 \| steps 200` | Wrong param count = wrong config; `grad_accum` ≠ 32 = wrong `micro_batch_size` |
| **Throughput** | **≥ ~100,000 tok/s** on a single H100 | See below |
| **Loss by step 100** | **falling, below ~7** (from ~10.4 at step 0) | Something is broken; do not proceed |

**On throughput:** ≥100k tok/s is the floor for the run to finish in a sane time,
and it corresponds to ~35% MFU (`flops_per_token × 100,000 ≈ 350 TFLOP/s`
against the H100's 989 TFLOP/s dense bf16 peak, using the same `6 × N + 12 ×
n_layer × d_model × seq_len` estimate the loop logs as `mfu`; see
[MFU](/docs/training#logging-throughput-and-mfu)). The **20-hour estimate
assumes ~139k tok/s (~49% MFU)**. These percentages are illustrative, not
measured: this repo has not yet run `d26` on real hardware; the run's own
`log.csv` `mfu` column is the actual figure once it has. If you are seeing 40k,
something is wrong:
most likely `torch.compile` silently disabled (check for the `torch.compile
disabled: not on CUDA` line) or fp32 instead of bf16 (`dtype: "auto"` picks bf16
only on CUDA). At 15 minutes you will have covered ~170-240 steps at these rates,
so step 100 is comfortably inside the window.

**On the loss:** step 0 must be ≈ **10.4** (`ln(32768) = 10.3972`, the uniform
baseline, see [Evaluation](/docs/evaluation#reading-the-number)). If it starts
somewhere else, initialization or the data is wrong. By step 100 (past the
250-step warmup's midpoint, so the LR is still ramping), it should be clearly
falling and under ~7. A flat loss at 10.4 means gradients are not flowing; a
`nan` means precision or data corruption.

Then throw the rehearsal away:

```bash
rm -rf out/rehearsal
```

<Callout type="warn">
  Use a **separate `--out-dir`** for the rehearsal. Sharing `out/d26` would leave a
  200-step `ckpt_last.pt` and a stale `log.csv` behind, and `log.csv` is opened
  with `"a"`, so the real run's curve would be appended to the rehearsal's.
</Callout>

## Step 5: Launch the real run

**In `tmux`.** Not optional: a dropped SSH connection kills a foreground process
and 20 hours with it.

```bash
tmux new -s train
uv run python -m tinyllm.train --config d26 \
    --tokenizer tokenizer/tokenizer.json
```

Detach with `Ctrl-b d`; reattach later with `tmux attach -t train`.

Optionally add Weights & Biases:

```bash
uv run python -m tinyllm.train --config d26 \
    --tokenizer tokenizer/tokenizer.json \
    --wandb tiny-llm
```

`--wandb` sets `wandb_project`; empty (the default) disables it entirely. The
master rank logs `train/loss`, `lr_scale`, `tok_per_sec`, `mfu` (when
computable: CUDA against a recognized GPU in `PEAK_FLOPS`, which an H100 is),
and `val/loss`, with the full model and train config captured as the run's
config.

### Multi-GPU

On an 8×H100 box, the same run via `torchrun`:

```bash
uv run torchrun --standalone --nproc_per_node=8 -m tinyllm.train \
    --config d26 --tokenizer tokenizer/tokenizer.json
```

Nothing else changes, and that is the design. `batch_tokens` is the invariant and
`grad_accum` absorbs the world size (`524,288 / (8 × 8 × 2048) = 4` instead of
32), so **the effective batch, the loss trajectory, and every hyperparameter are
identical**. You are buying wall-clock, not a different run. See
[Distributed](/docs/training/distributed).

## Step 6: Monitor

`out/d26/log.csv` is the source of truth: `step, split, loss, lr_scale,
tok_per_sec, mfu`, flushed on every write:

```bash
tail -f out/d26/log.csv
```

```bash
# just the validation curve
grep ',val,' out/d26/log.csv
```

Cadence for `d26`: train logs every **10** steps, val every **250**, checkpoint
every **1000**, sample every **1000**.

What to watch, in priority order:

- **Val loss falling.** The only number that matters. It should drop fast early,
  then grind. Expect ~3.5-4 by a few thousand steps and **2.7-3.0 at the end**.
- **`tok_per_sec` stable, `mfu` alongside it.** A gradual decline suggests
  thermal throttling or a noisy neighbor; a cliff suggests something else
  landed on the box. `mfu` is populated automatically on an H100 (it is in
  `PEAK_FLOPS`) and gives the same signal as a normalized percentage.
- **Loss spikes.** Occasional small ones are normal and
  [gradient clipping](/docs/training#gradient-clipping) (`grad_clip = 1.0`)
  is there to ride them out. A spike that does not recover within a few hundred
  steps, or a `nan`, means restart from the last checkpoint.
- **`lr_scale`.** Should be 1.0 through the plateau, then decay linearly over the
  final 20% (from step 15,258). If it is decaying early, `--steps` is wrong.

## Recovering from a preemption

This is why cheap preemptible instances are viable. Checkpoints are written
atomically to `out/d26/ckpt_last.pt` every 1,000 steps (~17 minutes), containing
model weights, **both optimizers' state**, the step, and both configs. To resume:

```bash
tmux new -s train
uv run python -m tinyllm.train --config d26 \
    --tokenizer tokenizer/tokenizer.json --resume
```

```
resumed from out/d26/ckpt_last.pt at step 12000
```

**The resume is bit-exact**, and that is verified rather than asserted: the
[smoke run's resume](/docs/runbooks/local-smoke#step-6-verify-resume) continued
from val 5.5012 to 5.4914 with no spike. It works because the optimizer state
rides along in the checkpoint (Muon's momentum buffers and AdamW's moments would
otherwise rebuild from zero, producing a visible bump) and because the data RNG is
a pure function of `(seed, rank, step, micro)` rather than a stateful generator:
step 12,000 draws the same batch whether it is reached by training or by
resuming. See [Checkpointing](/docs/training/checkpointing).

Three things to know:

- **`--resume` is not automatic.** Without the flag the run starts from step 0 and
  silently overwrites your checkpoint. This is the expensive mistake on this page.
- **You lose up to 1,000 steps** (~17 min, ~$1) per preemption: the work since
  the last checkpoint. Lower `ckpt_every` if your instance is evicted often.
- **`os.replace` is atomic**, so a crash *during* a checkpoint write leaves the
  previous checkpoint intact rather than a truncated file. There is no state in
  which you have no valid checkpoint.

## Step 7: Download the checkpoint

**Before you destroy the instance.** `ckpt_last.pt` is ~3.8 GiB (fp32 weights plus
Muon and AdamW optimizer state):

```bash
scp user@<host>:~/tiny-llm/out/d26/ckpt_last.pt ./ckpt_d26.pt
scp user@<host>:~/tiny-llm/out/d26/log.csv ./log_d26.csv
```

Take `log.csv` too; it is the run's entire history and it is 40 KB.

Then verify the checkpoint actually loads before you terminate anything:

```bash
uv run python -m tinyllm.sample --ckpt out/d26/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json --prompt "Photosynthesis is"
```

```bash
uv run python -m tinyllm.eval_hellaswag --ckpt out/d26/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json --limit 1000
```

## The cost table

| Setup | Throughput | Wall clock | ~$/h | **Total** |
|---|---|---|---|---|
| **1× H100 80GB** | ~139k tok/s | **≈20 h** | $2.5-3 | **≈$50-60** |
| **8× H100 (DDP)** | ~1.11M tok/s | **≈2.5 h** | $20-24 | **≈$50-60** |

Plus ~$5-9 for the 1-3 hour data-prep step at GPU rates, and ~$1 for the dress
rehearsal.

<Callout type="warn">
  **The 8×H100 figure is a linear-scaling estimate: it ignores DDP overhead.**
  It is 20 h ÷ 8, nothing more. Real 8-GPU throughput is lower: each optimizer step
  all-reduces ~489M gradients across the ranks, and although
  [`no_sync()`](/docs/training/distributed) restricts that to one collective per
  step rather than one per micro-step, it is not free. Expect somewhat more than
  2.5 hours in practice. The *total spend* is roughly a wash either way, which is
  the real point: **8× H100 buys wall-clock, not savings.**
</Callout>

Total spend either way is **~$60-70** including prep and rehearsal.

## Expected results

| Metric | Expected |
|---|---|
| Final val loss | **≈ 2.7-3.0** (perplexity ~15-20) |
| HellaSwag `acc_norm` | **≈ 0.30-0.35** (random = **0.25**) |
| Wall clock (1× H100) | ≈20 h |
| Cost | ≈$50-60 |

Both windows are wide because they should be; see
[Evaluation](/docs/evaluation#what-to-expect-at-this-scale) for how to read them.
The short version: val loss around 2.7-3.0 is only comparable against runs sharing
this tokenizer and data mix, and HellaSwag clearing chance by 5-10 points is a
**success** at 489M parameters. `acc_norm` at or below 0.26 after a full run means
something is broken; check the tokenizer/checkpoint pairing first, since evaluating
with a different `tokenizer.json` than the model trained on scores exactly chance.

Do not expect a chatbot. This is a base model trained on web text for one epoch,
with no instruction tuning and no RLHF. It completes documents. At 489M parameters
and 10B tokens it will produce fluent, largely coherent English with real but
shallow factual grounding: a large step up from the
[smoke run's](/docs/runbooks/local-smoke#sample-generations) confident nonsense
about photosynthesis, and a long way from a model you would ship.
