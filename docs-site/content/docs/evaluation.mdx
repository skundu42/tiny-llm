Two numbers say whether the run is working, and they say different things.
**Validation loss** is continuous, in-distribution, and available every 250 steps;
it is the number you watch. **HellaSwag** is discrete, downstream, and run
after the fact; it is the number that says the model learned something a human
would call knowledge. Both are implemented in this repo without a harness:
validation lives inside `tinyllm/train.py`, HellaSwag in
`tinyllm/eval_hellaswag.py`.

## Validation loss

### The protocol

Validation is a closure defined inside `run`, and every line of it matters:

```python
@torch.no_grad()
def eval_val() -> float:
    model.eval()
    loss_sum = torch.zeros((), device=device)
    for k in range(tc.val_batches):
        seed = [tc.seed, 999, k] if world == 1 else [tc.seed, 999, rank, k]
        rng = np.random.default_rng(seed)
        x, y = val_data.sample_batch(tc.micro_batch_size, mc.seq_len, rng, device)
        with ctx:
            _, loss = model(x, targets=y)
        loss_sum += loss.detach()
    model.train()
    local_mean = loss_sum / tc.val_batches
    return _distributed_mean(local_mean, world).item()
```

- **The batches are fixed.** In one process the seed is `[tc.seed, 999, k]`; under
  DDP it is `[tc.seed, 999, rank, k]`. It does not depend on the training step or
  wall clock. Every evaluation in a run scores **the exact same per-rank crops of
  the validation split**. This is the
  whole point: a val curve is only readable if the thing being measured does not
  move underneath it. If each eval drew fresh random crops, consecutive points
  would differ by sampling noise as well as by learning, and a 0.02 improvement
  would be unreadable. The `999` is a namespace constant making these draws
  independent from the training stream's `[seed, rank, step, micro]`; sampling
  with replacement still permits chance overlap.
- **`model.eval()` / `model.train()` bracket the call.** `tinyllm` has no dropout
  and no batchnorm, so this changes no math today; it is the correct habit, and
  it is what makes adding a stochastic layer later safe rather than silently
  wrong.
- **`@torch.no_grad()`**: no graph is built, so validation costs activation
  memory but no gradient memory.
- **The same autocast `ctx` as training.** Validation is measured in the same
  precision the model trains in, so the number is comparable to `loss_accum`
  rather than to a slightly different fp32 model.
- **The result is a plain mean over `val_batches` micro-batch losses.** Each
  `loss` is already a mean over its own tokens, so this is a mean-of-means; it
  is exactly a token-weighted mean only because every micro-batch has the same
  token count, which it does.

### How much data, and how often

`val_every` fires on `(step + 1) % tc.val_every == 0`, so with `d26`'s
`val_every = 250` the evals land at steps 249, 499, 749, …, and with `smoke`'s
`val_every = 100` they land at 99, 199, 299, … which is exactly where the
[recorded smoke numbers](/docs/runbooks/local-smoke) sit. It also fires
unconditionally on the last step, so a run always ends with a fresh val number
regardless of where the cadence landed.

| Preset | `val_batches` | `micro_batch_size` | `seq_len` | Tokens per eval |
|---|---|---|---|---|
| `d26` | 20 | 8 | 2048 | **327,680** |
| `smoke` | 8 | 8 | 512 | **32,768** |

Note what this is **not**: it is not a full pass over the validation split.
`d26`'s val split is ~10M tokens and each eval reads 328k of them, as 20 random
crops drawn with replacement. It is a fixed *sample*, not an exhaustive sweep:
cheap enough to run 76 times over a 19,073-step run without denting throughput,
and because the sample is frozen by the seed, its bias is a constant offset that
cancels out of every comparison you actually care about (step 249 vs step 19,072).

<Callout type="info">
  `eval_val()` is called on **every rank**, not just the master. Under DDP the val
  RNG includes `rank`, so each replica scores a different deterministic sample.
  The local means are all-reduced into one global mean and only the master logs
  it. This turns the parallel work into `world_size` times the validation
  coverage instead of redundantly scoring identical crops.
</Callout>

### Reading the number

The loss is a **per-token cross-entropy in nats** (`F.cross_entropy` on
`logits.float()`, natural log). Two conversions make it interpretable:

```
perplexity = exp(loss)
random-init loss = ln(vocab_size) = ln(32768) = 10.3972
```

Perplexity is the effective number of tokens the model is choosing between at
each position. A model that has learned nothing spreads probability uniformly
over the vocabulary and scores `ln(32768) = 10.3972` nats: perplexity 32,768,
the vocab size itself. That identity is the single best smoke test in the
project: the [recorded run](/docs/runbooks/local-smoke) opens at **10.4399**,
0.04 nats above the uniform baseline, which proves the
[initialization](/docs/architecture/embeddings-and-init) puts the logits where
they belong. It is asserted in the test suite as
`test_forward_loss_near_uniform_at_init`.

| Val loss (nats) | Perplexity | What it means |
|---|---|---|
| 10.40 | 32,768 | Untrained: uniform over the vocab |
| 5.50 | ~245 | The `smoke` run's final loss: English morphology and syntax, no facts |
| 3.0 | ~20 | Lower end of the `d26` expectation |
| 2.7 | ~15 | Upper end of the `d26` expectation |

**Expect `d26` to land at a val loss of roughly 2.7-3.0** after 19,073 steps over
~10B tokens. The window is wide on purpose: the exact figure moves with the
data mix and the tokenizer, and a val loss is only comparable against runs that
share both. A loss computed on a 32,768-token vocab is **not** comparable to one
published against a 50,257-token GPT-2 vocab; more vocab means more choices per
token and a mechanically higher cross-entropy. Compare `d26` against `d26`.

The shape matters more than the value. What you want to see is val tracking train
downward with no divergence between them. In the smoke run val ends at 5.5012
just *below* train's 5.5300: no overfitting, which is exactly right for a run
that consumes 9.8M tokens out of a 58M-token split and never sees an example
twice.

## HellaSwag

Validation loss cannot tell you whether the model knows that a person who picks
up a violin is more likely to play it than to eat it. HellaSwag can. Each example
is a context plus **four candidate endings**, one of which is the true
continuation and three of which are adversarially-generated distractors that are
grammatical and topical but wrong. The model never sees the labels and is never
trained on the task; it is scored purely on which ending it finds most likely.

This is the standard way to evaluate a base model: no prompt engineering, no
instruction following, no generation at all. Just likelihood.

### Scoring one example

```python
@torch.no_grad()
def ending_losses(model, ctx_ids, ending_ids_list, device) -> list[tuple[float, float]]:
    """Per ending: (sum, mean) cross-entropy over the ending tokens only."""
    out = []
    for ending in ending_ids_list:
        ids = torch.tensor([ctx_ids + ending], device=device)
        logits, _ = model(ids[:, :-1])
        targets = ids[:, 1:]
        losses = F.cross_entropy(
            logits.float().transpose(1, 2), targets, reduction="none"
        )[0]
        tail = losses[len(ctx_ids) - 1 :]  # positions predicting ending tokens
        out.append((tail.sum().item(), tail.mean().item()))
    return out
```

Four endings, four forward passes, each on the concatenation `ctx + ending`.

- **`reduction="none"`** is what makes this work. The default `F.cross_entropy`
  returns one scalar averaged over every position; here we need the per-position
  losses so we can throw most of them away. `logits.float().transpose(1, 2)`
  reshapes `(B, T, V)` to `(B, V, T)` because `cross_entropy` wants the class
  dimension second. The `[0]` unwraps the batch of one.
- **The slice is the whole trick.** `losses[j]` is the loss of predicting
  `ids[j+1]` from the prefix `ids[:j+1]`. The ending tokens are
  `ids[C] … ids[C+E-1]` where `C = len(ctx_ids)`, so the positions that predict
  them are `j = C-1 … C+E-2`, and `losses[len(ctx_ids) - 1:]` is exactly those
  `E` terms. **The context's own tokens contribute nothing to the score.** They
  must not: every candidate shares the same context, so including it would add an
  identical constant to all four: harmless for `acc`, but it would wreck
  `acc_norm`, where the four scores are divided by four *different* lengths.
  Note the first scored position, `C-1`, is the last *context* token: it is the
  position whose prediction is the ending's first token. An off-by-one here
  silently scores the wrong span.
- **Two reductions, returned together.** `tail.sum()` and `tail.mean()` are the
  raw and length-normalized log-likelihoods of the ending. The function computes
  both in one pass because the eval reports both.

In symbols, for an ending with tokens `e₁ … e_E` following context `c`:

```
sum  = Σᵢ −log P(eᵢ | c, e₁ … eᵢ₋₁)          # total surprisal of the ending
mean = sum / E                                # per-token surprisal
```

`sum` is the negative log-likelihood of the whole ending; `mean` is that divided
by its token count.

### acc vs acc_norm

```python
losses = ending_losses(model, ctx_ids, endings, device)
label = int(ex["label"])
acc += int(min(range(4), key=lambda i: losses[i][0]) == label)
acc_norm += int(min(range(4), key=lambda i: losses[i][1]) == label)
```

Both metrics are argmin over the four endings: lowest loss wins, since low loss
is high likelihood. They differ only in *which* of the two reductions they argmin
over:

- **`acc`** picks the ending with the lowest **summed** loss: the most probable
  ending, full stop.
- **`acc_norm`** picks the lowest **mean** loss: the most probable ending *per
  token*.

**Why normalize?** Because `sum` is biased toward short endings, and the bias is
structural rather than incidental. Every token contributes a strictly positive
surprisal (`−log P` of a probability below 1), so a longer ending accumulates a
larger total *no matter how good it is*. An 8-token ending at a comfortable 3.0
nats/token scores 24.0; a 4-token ending at a mediocre 5.0 nats/token scores
20.0, and wins. The model has not judged the short ending more plausible; it has
just had fewer chances to pay. Since HellaSwag's four candidates are natural
language of genuinely differing lengths, `acc` systematically rewards the short
one, and at small scale, where the model's per-token opinions are weak and
length dominates, that bias can be most of the signal.

`acc_norm` divides it out, which is why **`acc_norm` is the metric to quote** and
why it is usually the higher of the two for a base model. `acc` is kept because
the pair is diagnostic: a large gap between them tells you length is driving the
decisions.

<Callout type="warn">
  This `acc_norm` normalizes by **token count**. The widely-quoted
  `acc_norm` from EleutherAI's `lm-evaluation-harness` normalizes by the
  **character length** of the continuation string instead. The two are close in
  spirit and not identical in value, and ours additionally depends on our own
  32,768-token tokenizer. **Compare these numbers to other `tinyllm` runs, not to
  a published leaderboard.**
</Callout>

### The eval loop

```python
ds = load_dataset("Rowan/hellaswag", split="validation", streaming=True)
n = acc = acc_norm = 0
for ex in ds:
    if n >= args.limit:
        break
    ctx_ids = tok.encode(ex["ctx"])
    endings = [tok.encode(" " + e) for e in ex["endings"]]
    if any(len(ctx_ids) + len(e) > cfg.seq_len for e in endings):
        continue
    losses = ending_losses(model, ctx_ids, endings, device)
    label = int(ex["label"])
    acc += int(min(range(4), key=lambda i: losses[i][0]) == label)
    acc_norm += int(min(range(4), key=lambda i: losses[i][1]) == label)
    n += 1
    if n % 100 == 0:
        print(f"{n}: acc {acc / n:.4f} | acc_norm {acc_norm / n:.4f}")
```

- **The validation split**, streamed: HellaSwag's test labels are not public, so
  validation is the split everyone reports.
- **`tok.encode(" " + e)`**: the leading space is deliberate and load-bearing.
  Our [pre-tokenizer](/docs/tokenizer/training) binds a space to the word that
  follows it (` ?[^\s\p{L}\p{N}]+`, `[^\r\n\p{L}\p{N}]?\p{L}+`), so ` play` is a
  different token from `play`. The ending genuinely follows the context with a
  space in between, and encoding it without one would score a token sequence that
  never occurs in natural text.
- **Over-length examples are skipped, not truncated**, and `n` is not
  incremented for them, so the denominator counts only what was actually scored.
  Truncating the context would change the task; dropping the example is honest.
  At `seq_len = 2048` against HellaSwag's short contexts this effectively never
  fires; on the `smoke` preset's 512 it may.
- **`--limit` defaults to 1000** of the ~10k validation examples. Four forwards
  per example makes this the cost knob. 1,000 examples gives a standard error of
  roughly `√(0.3 × 0.7 / 1000) ≈ 1.4%`, so differences under ~3 points are
  noise; raise `--limit` before believing a small delta.

### Running it

```bash
uv run python -m tinyllm.eval_hellaswag \
    --ckpt out/d26/ckpt_last.pt \
    --tokenizer tokenizer/tokenizer.json \
    --limit 1000
```

`--device` is optional and auto-selects `cuda` → `mps` → `cpu`. Output is a
progress line every 100 examples and a final summary. **No `d26` run has been
executed; the line below illustrates the format, not a measurement:**

```
# illustrative format: no d26 run exists yet; these are not measured values
final (1000 examples): acc 0.2880 | acc_norm 0.3240
```

### What to expect at this scale

| Model | HellaSwag `acc_norm` |
|---|---|
| Random guessing | **0.25** |
| `d26` (489M params, 10B tokens) | **≈ 0.30-0.35** |
| Humans | ~0.95 |

**Four choices means chance is 0.25, and that is the number to beat.** A `d26`
run landing at **0.30-0.35** is a success. That gap over chance looks small next
to a frontier model's 0.85+, and it is exactly what 489M parameters on 10B tokens
buys. HellaSwag was built to be adversarially hard for models of roughly this
generation, and clearing chance by 5-10 points means the model has real, if thin,
commonsense preference. It is not a rounding error: at 1,000 examples, 0.32 vs
0.25 is ~5 standard errors.

Anything at or below **0.26** after a full run means something is broken, most
likely a tokenizer/checkpoint mismatch (an eval run with a different
`tokenizer.json` than the model trained on scores exactly chance, because the
token ids are meaningless to the model). Check that first.

And a caution worth stating plainly: **HellaSwag is not sensitive at this scale.**
Between 0.30 and 0.33 you are reading noise as much as capability. Use val loss
to steer the run and HellaSwag as an end-of-run check that the loss corresponds
to something outside the training distribution, not as a hill to climb.
